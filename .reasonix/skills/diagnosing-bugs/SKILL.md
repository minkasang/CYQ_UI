# 科学调试法

一套用于解决复杂 bug 的纪律性方法。核心原则：**反馈环是一切**——没有一个紧致的红/绿信号，看再多代码也没用。

> 如果项目有 `CONTEXT.md`，先读取它来建立相关模块的心智模型；检查相关区域的 ADR。

---

## 阶段 1：建立反馈环（最关键！）

> **这一阶段就是整个 skill 的核心。** 其余步骤是机械操作。如果你有一个**紧致**的红/绿信号——一个能捕获**这个** bug 的信号——你就能找到原因。没有它，盯着代码看再多也没用。

**在这里投入不成比例的精力。要有进攻性，要有创造力。拒绝放弃。**

### 构建反馈环的 10 种方法（按推荐顺序尝试）

1. **失败测试**：在任何能接触到 bug 的缝上写一个失败的测试（单元/集成/E2E）
2. **HTTP 脚本**：对运行的开发服务器发起 curl/HTTP 请求
3. **CLI 调用**：用固定输入调用，与已知正确的快照对比
4. **无头浏览器脚本**：Playwright/Puppeteer 驱动 UI，断言 DOM/控制台/网络
5. **重放捕获的 trace**：把真实网络请求/事件日志存到磁盘，隔离重放
6. **一次性 harness**：启动系统的最小子集，用一个函数调用执行 bug 代码路径
7. **属性/模糊测试循环**：如果 bug 是"有时候输出错误"，运行 1000 次随机输入
8. **二分 harness**：如果 bug 在两个已知状态之间出现，自动化"从状态 X 启动，检查，重复"
9. **差异循环**：对同一输入运行旧版本 vs 新版本，diff 输出
10. **HITL 脚本**：最后手段。如果必须有人手动点击，用结构化脚本来驱动

### 收紧反馈环

一旦有了**一个**环，立即**收紧**它：

- 能不能更快？（缓存 setup、跳过无关初始化、缩小测试范围）
- 能不能让信号更精准？（断言具体症状，不是"没崩溃"）
- 能不能更确定性？（固定时间、固定随机种子、隔离文件系统）

**30 秒的不可靠循环和没有循环差不多；2 秒的确定性循环是调试超能力。**

### 非确定性 bug

目标不是干净的复现，而是**更高的复现率**。循环触发 100 次，并行化，加压力，缩小时间窗口。50% 的 flaky bug 可以调试；1% 不行——持续提高复现率直到可调试。

### 实在无法构建反馈环时

停下来明确说清楚。列出你尝试了什么。向用户请求：(a) 复现环境访问权，(b) 捕获的产物（HAR 文件、日志 dump、core dump、带时间戳的屏幕录制），或 (c) 添加临时生产环境检测的权限。**在没有反馈环的情况下不要进入假设阶段。**

### 阶段 1 完成标准

反馈环满足以下全部条件：

- [ ] **能变红**：它执行真正的 bug 代码路径，断言用户的**确切症状**，能在这个 bug 上变红、修复后变绿
- [ ] **确定性**：每次运行同样结果（flaky bug：足够高的复现率）
- [ ] **快**：秒级，不是分钟级
- [ ] **AI 可运行**：你可以无人值守运行它

---

## 阶段 2：复现并最小化

运行反馈环。确认它变红——bug 出现了。

### 确认

- [ ] 反馈环产生了**用户描述的**失败模式——不是碰巧在附近的另一个失败
- [ ] 失败在多次运行中可复现
- [ ] 你已捕获确切症状（错误信息、错误输出、慢的时间），以便后续阶段验证修复

### 最小化

一旦变红，把复现缩小到**仍能变红的最小场景**。一次切掉一个输入、调用者、配置、数据、步骤——每切一刀后重新运行。

**为什么**：最小复现缩小了阶段 3 的假设空间，并成为阶段 5 的干净回归测试。

完成标准：**每个剩余元素都是不可或缺的**——去掉任何一个都会让环变绿。

---

## 阶段 3：提出假设

在测试**任何**假设之前，先生成 **3-5 个排序假设**。只生成一个假设会锚定在第一个看起来合理的想法上。

每个假设必须**可证伪**：

```
格式："如果 <X> 是原因，那么 <改 Y> 会让 bug 消失 / <改 Z> 会让它更糟。"
```

如果你说不出预测，这个假设就是个感觉——丢弃或收紧它。

**在测试前把排序列表展示给用户。** 用户通常有领域知识可以瞬间重新排序，或知道已经排除的假设。不阻塞在用户反馈上——如果用户 AFK，按你的排序继续。

---

## 阶段 4：定位（Instrumentation）

每次探测必须对应阶段 3 的一个具体预测。**一次只改一个变量。**

工具偏好：

1. **调试器 / REPL 检查**：一个断点胜过十个日志
2. **定向日志**：在能区分假设的边界处打日志
3. 绝不"全都打日志然后 grep"

**每个调试日志用唯一前缀标记**，如 `[DEBUG-a4f2]`。最后清理时一个 grep 搞定。没标记的日志保留；有标记的全部清理。

**性能分支**：对于性能退化，日志通常是错误工具。取而代之：建立基线测量（计时 harness / `performance.now()` / profiler / 查询计划），然后二分。先测量，后修复。

---

## 阶段 5：修复 + 回归测试

在修复**之前**写回归测试——但只存在**正确的缝**时才写。

**正确的缝**是指测试能够以 bug 在调用点发生的**真实模式**来执行。如果唯一可用的缝太浅（单调用者测试但 bug 需要多调用者，单元测试无法复现触发链），在那里写回归测试会带来虚假信心。

**如果没有正确的缝存在，这本身就是一个发现。** 记录下来。代码库架构正在阻止这个 bug 被锁定。标记这一点，交给 `/improve-codebase-architecture`。

如果存在正确的缝：

1. 把最小化复现变成该缝上的失败测试
2. 观察它失败
3. 应用修复
4. 观察它通过
5. 用原始（未最小化的）场景重新运行阶段 1 反馈环

---

## 阶段 6：清理 + 复盘

声明完成前必须：

- [ ] 原始复现不再出现（重新运行阶段 1 反馈环）
- [ ] 回归测试通过（或缝的缺失已记录）
- [ ] 所有 `[DEBUG-...]` 已移除（grep 前缀确认）
- [ ] 一次性原型已删除（或移到明确标记的调试位置）
- [ ] 被证实正确的假设写在 commit/PR 信息中——让下一个调试者学习

**然后问：什么能防止这个 bug？** 如果答案涉及架构变化（没有好的测试缝、纠缠的调用者、隐藏耦合），把具体情况交给 `/improve-codebase-architecture`。在修复**之后**做这个建议，不是之前——你现在比开始时掌握了更多信息。

---

## 禁止行为

- ❌ 反馈环还没构建就开始看代码猜测原因
- ❌ 只生成一个假设就开始测试
- ❌ 一次改多个变量导致无法确定哪个是原因
- ❌ 打一大堆无标签日志然后用 grep 找
- ❌ 在没有正确缝的地方强行写回归测试（虚假信心）
- ❌ 调试日志留在代码中不清理
