# 测试验证纪律

## 核心原则

1. **每阶段必验证**：代码变更后必须有验证步骤，确认正确后再交付
2. **验证分层**：不同层级用不同类型的测试覆盖
3. **验证不做假**：验证通过意味着真正验证过，不能跳过或降级
4. **自适应技术栈**：命令和工具根据项目实际技术栈选择，不预设特定框架

---

## TDD 开发模式：红 → 绿 → 重构

当**构建新功能或修复 bug** 时，使用红绿重构循环。这是"开发方式"，上面的验证门禁是"质量保障"——两者互补。

### 核心理念

- **测试应该通过公共接口验证行为，不是验证实现细节。** 代码可以完全改变，测试不应该变。
- **好的测试读起来像规格说明**："用户可以 checkout" 告诉你什么能力存在。这些测试在重构中存活，因为它们不关心内部结构。
- **坏的测试耦合于实现**：mock 内部协作者、测试私有方法。警告信号：重构时测试失败，但行为没变。

> 如果项目有 `CONTEXT.md`，测试名称和接口词汇应与项目的领域语言一致。

### 垂直切片 vs 水平切片（关键！）

```
❌ 水平切片（错误）：            ✅ 垂直切片（正确）：
   RED:  test1,test2,test3         RED→GREEN: test1→impl1
   GREEN: impl1,impl2,impl3        RED→GREEN: test2→impl2
                                   RED→GREEN: test3→impl3
```

**为什么水平切片产生烂测试**：
- 批量写的测试验证**想象**的行为，不是**实际**行为
- 你测试的是数据结构和函数签名的**形状**，不是用户可见的行为
- 测试对真正的变化不敏感——行为坏了测试还能过，行为没变测试反而失败
- 你在理解实现之前就锁定了测试结构

**垂直切片 = tracer bullet（曳光弹）**：每个切片是一条贯穿所有层的完整路径（schema → 逻辑 → UI → 测试），独立可验证、可 demo。

### 红绿重构循环

#### 1. 规划

动手前先确认：
- [ ] 需要哪些接口变更？
- [ ] 哪些行为最重要需要测试？（优先排序——你不可能测试一切）
- [ ] 有没有深化模块的机会？（参考 healthy-architecture 1.4 深度模块设计）

#### 2. Tracer Bullet（第一颗曳光弹）

写**一个**测试确认**一件事**：

```
RED:   写第一个行为的测试 → 测试失败
GREEN: 写最少代码通过 → 测试通过
```

这证明了整条路径从端到端可以工作。

#### 3. 增量循环

对每个剩余行为：

```
RED:   写下一个测试 → 失败
GREEN: 最少代码通过 → 通过
```

规则：
- 一次一个测试
- 只写够让当前测试通过的代码
- 不要预测未来的测试
- 保持测试聚焦于可观察行为

#### 4. 重构

所有测试通过后，寻找重构机会：
- [ ] 抽取重复
- [ ] 深化模块（把复杂度移到简单接口背后）
- [ ] 自然地应用 SOLID
- [ ] 每次重构后运行测试

**绝不在 RED 状态重构。** 先到 GREEN。

### 每轮检查清单

```
[ ] 测试描述行为，不是实现
[ ] 测试只使用公共接口
[ ] 测试能在内部重构中存活
[ ] 代码对于这个测试来说是最少的
[ ] 没有添加猜测性功能
```

## 验证金字塔

```
          /\
         /端到端\       少量，验证关键用户流程
        /--------\
       /  集成测试 \     中等，验证模块间交互
      /------------\
     /   单元测试    \   大量，验证纯逻辑、边界条件
    /----------------\
```

| 层级 | 占比建议 | 覆盖内容 | 执行频率 |
|------|----------|----------|----------|
| **单元测试** | ~60% | 纯逻辑、工具函数、边界条件、错误路径 | 每次提交 |
| **集成测试** | ~30% | 模块间接口、数据流贯通、外部依赖交互 | 每次 PR |
| **端到端测试** | ~10% | 关键用户操作流程、核心业务闭环 | 每次发布 |

## 验证流程

### 每阶段必须执行的验证步骤（按顺序）

1. **编译 / 语法检查**
   - 代码能编译通过，无语法错误
   - 类型检查通过（如项目有对应工具）
   - **命令检测策略**：先检查项目已有的构建脚本（package.json、Makefile、Cargo.toml、pom.xml、build.gradle 等），运行对应的构建命令；如果没有，使用语言标准命令

2. **Lint 检查**
   - 代码风格符合项目规范
   - 无未使用的变量、导入
   - **命令检测策略**：检查项目已配置的 lint 工具（如 .eslintrc、pyproject.toml、.golangci.yml、checkstyle.xml 等），运行项目自带的 lint 脚本；如果项目未配置 lint，可跳过此步

3. **单元测试**
   - 新增代码的关键路径、边界条件、错误处理必须有测试覆盖
   - 已有测试必须全部通过，不能有退化
   - **命令检测策略**：检查项目测试脚本（如 `npm test`、`cargo test`、`go test ./...`、`pytest`、`mvn test`、`gradle test` 等），运行对应的测试命令

4. **集成检查**
   - 确认该阶段的变更没有破坏已有功能
   - 运行受影响的集成测试

5. **手动验证（必要时）**
   - 无法自动化验证的 UI、交互等
   - 给出明确的验证步骤

**只有以上全部通过后，才能进入下一阶段。**

### 快速命令参考（按技术栈）

| 技术栈 | 构建检查 | Lint 工具 | 测试命令 |
|--------|----------|-----------|----------|
| Node.js | `npm run build` | ESLint / Prettier | `npm test` |
| TypeScript | `tsc --noEmit` | ESLint / Prettier | `npm test` |
| Python | `python -m py_compile` | flake8 / ruff / pylint | `pytest` |
| Go | `go build ./...` | golangci-lint / go vet | `go test ./...` |
| Rust | `cargo build` / `cargo check` | cargo clippy / rustfmt | `cargo test` |
| Java | `mvn compile` / `gradle build` | checkstyle / spotbugs | `mvn test` / `gradle test` |
| C/C++ | `cmake --build` / `make` | cpplint / clang-tidy | `ctest` / `make test` |
| C#/.NET | `dotnet build` | dotnet format / StyleCop | `dotnet test` |
| Ruby | `ruby -c` | rubocop / standard | `rspec` / `rake test` |
| Dart/Flutter | `dart analyze` / `flutter analyze` | dart format | `flutter test` / `dart test` |

## 测试编写规范

### 单元测试要求

- **覆盖率目标**：关键路径尽可能全覆盖，整体覆盖率根据项目阶段设定（早期项目可放宽，生产代码应 >80%）
- **测试独立**：每个测试不依赖其他测试的执行顺序
- **命名清晰**：遵循语言社区的命名惯例（Python: `test_<模块>_<场景>`, Java: `<场景>Test`, JavaScript: `test('<场景>')` 等）
- **AAA 模式**：Arrange（准备）→ Act（执行）→ Assert（断言）

### 必须测试的场景

| 场景 | 说明 |
|------|------|
| **正常路径** | 正常输入，得到预期输出 |
| **边界条件** | 空值、零值、最大值、最小值 |
| **异常路径** | 非法输入、外部依赖失败、超时 |
| **并发安全** | 如果有共享状态，测试并发读写 |

### 集成测试要求

- 验证模块间接口契约是否一致
- 验证数据在各模块间流动的正确性
- 外部依赖（数据库、API）用真实或容器化的环境，非 mock

## 验证门禁

### 新代码的验证门禁

```
  提交代码
     ↓
  编译检查 ──失败──→ 修复后重新提交
     ↓ 通过
  Lint 检查 ──失败──→ 修复后重新提交
     ↓ 通过
  单元测试 ──失败──→ 修复后重新提交
     ↓ 通过
  集成检查 ──失败──→ 修复后重新提交
     ↓ 通过
  ✓ 进入下一阶段
```

### 阶段间验证门禁

```
  阶段 N 完成
     ↓
  阶段 N 的测试全部通过？
     ↓ 是                ↓ 否
  已有测试无退化？        修复 →
     ↓ 是
  ✓ 允许进入阶段 N+1
```

---

## 禁止行为

- ❌ 跳过验证直接进入下一阶段
- ❌ 忽略测试失败继续推进
- ❌ 写了测试但用 skip / 注释掉
- ❌ 仅靠手动测试替代自动化测试
- ❌ 为了覆盖率指标写无意义的测试
- ❌ 集成测试全部用 mock（失去集成测试的意义）
- ❌ 验证步骤只说不做
