# 事前对齐盘问（Grilling）

**核心理念**：软件项目最常见的失败模式是对齐失败。你以为 AI 知道你要什么，等看到产物才发现它完全没理解你。修复这个问题的办法是**盘问会话**——让 AI 对你进行深度提问，遍历设计决策树的每个分支，直到所有模糊点都被解决，才开始写代码。

> 来自 David Thomas & Andrew Hunt《程序员修炼之道》："没有人确切知道自己想要什么。"

---

## 盘问原则

### 基本规则

1. **一次一个问题**：逐个提问，等待用户对每个问题给出反馈后再继续。一次问多个问题会让人困惑。
2. **提供建议答案**：每个问题给出你的推荐答案和理由，让用户只需确认或修正。
3. **走完每个分支**：对设计树做深度优先遍历，在一个分支被完全解决之前不跳转到另一个分支。
4. **能用代码探索回答的就不问**：如果问题可以通过阅读代码库来回答，直接去读代码而不是问用户。

### 盘问维度

根据计划的不同类型，盘问应覆盖以下维度（不限于此）：

| 维度 | 示例问题 |
|------|---------|
| **范围** | "这到底要改变什么？哪些不动？" |
| **用户路径** | "用户从哪开始？经过哪些步骤到达结果？" |
| **数据流** | "数据从哪来？经过哪些模块？最终写到哪里？" |
| **接口** | "哪些接口需要改变？新增还是修改？调用者是谁？" |
| **边界条件** | "空状态长什么样？错误状态怎么处理？极限情况？" |
| **依赖** | "这东西依赖什么？什么依赖它？顺序是怎样的？" |
| **优先级** | "这些功能里面，哪个是必须有的？哪个可以后面再加？" |
| **验收标准** | "做完后你拿什么标准判断它是对的还是错的？" |

### 与领域建模联动

盘问过程中，当术语确定或决策形成时，**在对话中即时**维护项目知识：

1. **术语确定时**：立即写入 `CONTEXT.md`。如果文件不存在，在第一个术语确定后创建它
2. **做出不可轻易逆转的决策时**：提议创建 ADR。只在满足三条件时提议（难以逆转 + 没有上下文会惊讶 + 真正的权衡结果）
3. **收紧模糊语言**：当用户使用模糊或过载的术语时，提出精确的规范术语
4. **与术语表对照**：当用户使用的术语与已有 `CONTEXT.md` 冲突时，立即指出

## 盘问流程

### 阶段 1：理解意图

从用户描述中提取核心目标：
- "我理解你想做 X。让我确认一下——这是不是意味着 Y 和 Z 也包含在内？"
- 用一句话概括你理解的范围，让用户确认或修正

### 阶段 2：遍历决策树

对设计空间进行深度优先遍历：

1. 找到一个悬而未决的决策点
2. 提出一个具体问题，附上推荐答案
3. 等待用户反馈
4. 根据反馈，要么深入该分支的子决策，要么回到上一层
5. 重复直到所有分支都被解决

### 阶段 3：产出共识总结

当用户表示满意或所有分支已穷尽时，输出一份简洁的共识总结：

```markdown
## 我们达成的共识

### 要做什么
- 要点 1
- 要点 2

### 不做什么（明确排除）
- 要点 1

### 关键决策
- 决策 A：我们选了 X 而不是 Y，因为 Z
- 决策 B：...

### 验收标准
- [ ] 标准 1
- [ ] 标准 2

### 涉及的模块/文件
- 模块 A（变更类型：新增/修改/不变）
- 模块 B
```

## 禁止行为

- ❌ 盘问还没结束就开始写代码
- ❌ 一次抛出多个问题
- ❌ 用户说"差不多就这样"后不输出共识总结
- ❌ 跳过显而易见的决定——看起来"明显"的事往往是最大的误解来源
- ❌ 在盘问中评判用户的选择——盘问的目的是对齐理解，不是质疑设计
- ❌ 忘记在盘问过程中更新 CONTEXT.md
