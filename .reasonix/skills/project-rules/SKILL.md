# AI 执行规则

所有任务的状态和阶段变更实时记录在 `.reasonix/task_manifest.yaml`，AI 启动时应先读此文件了解全局进度，无需遍历各任务目录。

## 任务清单规则
- 每个任务对应 `共享执行规则文件/{任务名}/`，产出 tasks.md / architecture.md / test.md
- 一次只能 1 个 in_progress
- 中途插新需求时，当前任务变 blocked，新任务为 in_progress，`parent`/`blocked_by` 关联，支持多层嵌套
- 完成后移入 archived 段
- **每次 AI 启动读完 manifest 后，必须校对所有 active 任务状态是否与实际情况一致，不一致则立即更新 manifest**

## 任务拆解规则

多级任务分解：执行任务时必须拆解为多级子任务，拆到每一步可独立完成、可独立验证、可独立回退。拆解层级和步数由任务复杂度动态决定，不固定。每步完成后必须进行测试，通过后进入下一步。

关键原则：
1. 任务拆解要考虑任务复杂度、依赖关系、时间限制
2. 不能一口气做完所有阶段——如果任务很大，必须分多次对话来做
3. 优先保证质量、架构健康性、代码完整性，不追求速度
4. 拆解产出文件放在 `共享执行规则文件/任务名称/tasks.md`

## 架构设计规则

架构设计必须健康，遵守以下质量目标：

| 质量属性 | 说明 | 相关设计维度 |
|----------|------|-------------|
| **可靠性** | 系统稳定运行，错误可恢复 | 错误处理、数据流 |
| **可维护性** | 代码易于理解、修改、扩展 | 项目结构、接口与契约 |
| **可扩展性** | 新增功能时修改最小化 | 设计原则、接口与契约 |
| **性能效率** | 响应快速，资源占用合理 | 数据流、可测试性 |
| **可测试性** | 核心逻辑可独立测试 | 可测试性设计 |
| **可观测性** | 系统状态可监控、可追踪 | 错误处理、日志 |
| **安全性** | 数据安全，权限可控 | 接口与契约、错误处理 |
| **兼容性** | 向后兼容，版本平滑升级 | 接口与契约 |
| **可移植性** | 跨平台、跨环境部署 | 项目结构 |

实现手段遵循 healthy-architecture skill 的 6 大设计维度：
1. 设计原则：SOLID、组合优于继承、最少知识原则
2. 项目结构：按功能分包、三次法则
3. 接口与契约：小接口、参数明确、向后兼容
4. 错误处理：fail-fast vs 容错降级、错误传播链
5. 数据流与状态管理：单向数据流、不可变数据优先
6. 可测试性设计：依赖注入、副作用隔离

## 测试规则

测试分类：操作间干扰、数据一致性、边界与极限、恢复能力、用户预期管理、渲染与视觉、交互反馈、性能与流畅度、安全与权限、可维护性、实际效果。

测试策略三层：单元测试（纯逻辑）、集成测试（模块交互）、端到端测试（用户流程）。

## 完整开发流程

```
拆解 → 设计 → 编码 → 测试 → 验证通过 → 进入下一步
```

每个阶段产出对应文件（tasks.md / architecture.md / 代码 / test.md），完成后对照健康检查清单和禁止行为清单自查。

## 禁止行为

- ❌ 没有接口设计就直接写实现
- ❌ 为了"更快完成"而牺牲架构清晰度
- ❌ 深层继承链（>3 层）或用继承复用行为
- ❌ 链式调用穿过 3 个以上对象
- ❌ 第一次出现重复就急着重构抽象
- ❌ 吞异常不传播上下文
- ❌ 混放业务异常和技术异常
- ❌ 状态放在共享层但只被一个模块使用
