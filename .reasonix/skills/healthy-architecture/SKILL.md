# 架构设计规范与最佳实践

## 与项目规则的关系

本 skill 是 `AGENTS.md` 中架构设计规则的实现手段补充：

- **rules 定义目标**：9个质量属性（可靠性、可维护性、可扩展性、性能效率、可测试性、可观测性、安全性、兼容性、可移植性）
- **skill 提供手段**：6大设计维度（设计原则、项目结构、接口与契约、错误处理、数据流与状态管理、可测试性设计）

**使用方式**：
1. 先阅读 rules 了解质量目标
2. 再参考本 skill 了解如何实现这些目标
3. 在架构设计文档中明确说明设计维度如何支撑质量目标

## 核心原则

1. **模块职责清晰**：高内聚低耦合，每个模块边界明确
2. **接口先行**：先定义接口契约，再实现细节
3. **避免过度工程**：不为尚未出现的需求预留扩展点
4. **保证可测试性**：核心逻辑不依赖全局状态，可独立测试
5. **语言无关**：以下原则适用于任何编程语言（OOP、FP、过程式），请根据项目语言特性适配

## 六大设计维度

### 一、设计原则

#### 1.1 SOLID 原则（OOP 语言）及等价原则（非 OOP 语言）

| OOP 原则 | 通用等价原则 | 说明 |
|----------|-------------|------|
| **单一职责（SRP）** | 一模块一事 | 一个模块/函数只负责一件事，只有一个变更原因 |
| **开闭原则（OCP）** | 扩展不加改 | 新增功能时添加新代码，而非修改已有稳定代码 |
| **接口隔离（ISP）** | 小接口优于大接口 | 调用方不应依赖它不需要的东西（大 struct 拆小 struct，大 protocol 拆小 protocol） |
| **依赖反转（DIP）** | 依赖抽象不依赖实现 | 高层逻辑依赖接口/trait/protocol，不依赖具体实现 |

> 注意：里氏替换（LSP）仅适用于继承体系，非 OOP 语言（Go、Rust、C）无需关注。用组合替代继承可自然规避此问题。

#### 1.2 组合优于继承

- 优先使用组合来复用行为，而非深层继承链
- 在 Go 中用 struct 嵌入，Rust 中用组合 + trait，C 中用函数指针表
- 只有明确的 "is-a" 关系才用继承（仅限于有继承机制的语言）
- 深层继承链（>3 层）几乎总可以用组合重构成更好的设计

#### 1.3 最少知识原则（Law of Demeter）

- 一个模块应对其他模块有尽可能少的了解
- 方法只应该调用：自身、方法参数、自身创建的依赖、直接组件
- 禁止链式调用穿过多个对象：`a.getB().getC().doSomething()` → 封装在中间层

#### 1.4 深度模块设计（Deep Modules）

> 来自 John Ousterhout《A Philosophy of Software Design》与 Matt Pocock 的实践总结。

**核心概念**：好的模块应该是**深的**——用一个小接口隐藏大量实现。

```
深度模块（推荐）：              浅度模块（避免）：
┌─────────────────────┐      ┌───────────────────────────┐
│  小接口（少量方法）   │      │  大接口（很多方法/参数）    │
├─────────────────────┤      ├───────────────────────────┤
│                     │      │                           │
│  深实现（复杂逻辑）   │      │  薄实现（只是透传）         │
│                     │      │                           │
└─────────────────────┘      └───────────────────────────┘
```

**术语表**（设计讨论中使用这些精确词汇，不要换成"组件""服务""API"等模糊词）：

| 术语 | 定义 |
|------|------|
| **模块（Module）** | 任何有接口和实现的东西——函数、类、包都可以是模块 |
| **接口（Interface）** | 调用者需要知道的**所有**信息：类型签名、不变量、顺序约束、错误模式、性能特征 |
| **实现（Implementation）** | 模块内部的东西，接口背后的代码体 |
| **深度（Depth）** | 接口背后的行为量 / 调用者需要学习的接口量。深度模块 = 大行为 + 小接口 |
| **缝（Seam）** | 模块接口所在的位置——可以在不编辑该处代码的情况下改变行为的地方 |
| **适配器（Adapter）** | 在缝上满足接口的具体东西。描述**角色**（填充哪个槽），不是实质 |
| **杠杆（Leverage）** | 调用者从深度中获得的：每个接口单位换取更多能力 |
| **局部性（Locality）** | 维护者从深度中获得的：变更、bug、知识、验证集中在同一处 |

**核心原则**：

1. **删除测试**：想象删除这个模块。如果复杂度消失了，它就是个透传（没用）。如果复杂度分散到 N 个调用者，它是在赚它的位置。

2. **接口就是测试面**：调用者和测试跨越同一个缝。如果你想测到接口**背后**去，说明模块形状可能不对。

3. **一个适配器 = 假想缝，两个适配器 = 真正的缝**：不要引入缝除非确实有东西在它上面变化。单一适配器的缝只是间接层。

4. **内部缝 vs 外部缝**：深度模块可以有内部缝（私有于实现，给自己测试用），以及外部缝（在接口处）。不要因为测试用就把内部缝暴露到接口上。

**设计时自问**：
- 能不能减少方法数量？
- 能不能简化参数？
- 能不能在接口背后隐藏更多复杂度？

**深化（Deepening）策略**：当发现一组浅度模块时，按依赖分类决定如何深化：

| 依赖类别 | 示例 | 深化策略 |
|---------|------|---------|
| **进程内** | 纯计算、内存状态 | 直接合并模块，通过新接口测试 |
| **本地可替换** | PGLite 替代 Postgres | 用本地替代品测试，缝是内部的 |
| **远程但自有** | 微服务、内部 API | 定义端口（接口），注入适配器，测试用内存适配器 |
| **真正外部** | Stripe、Twilio 等 | 作为注入端口接收，测试用 mock 适配器 |

**注意**：深化后，旧的浅度模块单元测试成为废料——删除它们。在深化后的模块接口上写新测试。测试断言接口的可观察输出，不是内部状态。

### 二、项目结构

#### 2.1 按功能分包 vs 按类型分包

| 方式 | 适用场景 | 通用示例 |
|------|----------|----------|
| **按类型** | 小型项目，关注点固定 | 按技术分层：`handlers/`, `models/`, `utils/` |
| **按功能** | 中大型项目，多人协作 | 按业务域：`auth/`, `payment/`, `user/` |
| **混合** | 大型项目 | 功能内按技术分层，公共部分单独管理 |

**选择标准**：当某个功能模块需要独立测试、独立部署或多人并行开发时，按功能分包。

各语言社区都有惯用的项目布局（Go 的 `cmd/` / `internal/` / `pkg/`，Rust 的 workspace 成员，Python 的 `src/` 布局等），优先遵循社区惯例。

#### 2.2 公共代码抽取时机 — 三次法则

- 第一次出现：直接写，不抽象
- 第二次出现：可以容忍重复
- **第三次出现：才抽取公共代码**

过早抽象比重复代码危害更大——错误的抽象会固化在架构中，后续改造成本极高。

#### 2.3 目录命名与文件组织

- 命名风格与项目已有风格保持一致（kebab-case、snake_case、PascalCase 均可，关键是统一）
- 每层目录有明确的职责边界，不要混放不同层级的代码
- 优先遵循语言社区的目录命名惯例

### 三、接口与契约

#### 3.1 接口设计规范

- **小接口优于大接口**：调用方不应依赖它不需要的方法/字段（Go 的 interface 天然鼓励小接口；Rust 的 trait 也应保持精简）
- **参数明确**：避免 bool 标志参数分裂行为，改用独立函数或枚举
- **返回值一致**：同一层级的返回值模式统一（Go: `(result, error)`，Rust: `Result<T, E>`，Python: 统一异常体系等）
- **命名自解释**：接口名和参数名应让调用者不看文档就能理解

#### 3.2 向后兼容策略

| 策略 | 说明 |
|------|------|
| **加新不删旧** | 新增接口参数/字段，标记旧版为 deprecated，等调用方迁移完再移除 |
| **版本化** | 重大变更时引入 v2 接口，v1 保留一段时间 |
| **默认值** | 新增参数提供合理默认值，保证旧调用方不受影响 |

#### 3.3 输入校验归属

- **外部边界校验**：用户输入、API 请求、文件读取——在最外层严格校验
- **内部信任**：模块间调用做轻量断言（assert / precondition），不重复校验
- **校验逻辑归属**：公共库/工具函数在被调用方校验；业务逻辑在调用方控制

### 四、错误处理

#### 4.1 fail-fast vs 容错降级

| 策略 | 适用场景 | 各语言示例 |
|------|----------|-------------|
| **fail-fast** | 开发阶段、配置缺失、前置条件不满足 | CI pipeline、`panic()`(Go)、`assert!`(Rust) |
| **容错降级** | 生产环境、非核心功能、外部依赖不稳定 | 重试机制、断路器、fallback 默认值 |

**选择标准**：系统启动和关键路径用 fail-fast；用户交互过程中非关键功能用容错降级。

#### 4.2 错误传播链

- 不要吞异常/错误，每一层捕获时包装上下文信息
- 错误信息应包含：发生了什么、为什么、业务上下文（哪个用户/哪个操作）
- 利用语言特性：Go 的 `fmt.Errorf("xxx: %w", err)`、Rust 的 `anyhow::Context`、Java 的 `cause chain`、Python 的 `raise ... from e`

#### 4.3 业务异常 vs 技术异常

| 分类 | 示例 | 处理方式 |
|------|------|----------|
| **业务异常** | 余额不足、库存为0、权限不足 | 定义错误码/异常类型，上层展示友好提示 |
| **技术异常** | 网络超时、数据库连接失败 | 统一兜底，记录日志，告警 |

业务错误和技术错误类型体系分开定义，不要混用。

### 五、数据流与状态管理

#### 5.1 单向数据流

- 数据源唯一，变更可追溯
- 子模块不直接修改上游传来的数据，通过回调/事件/消息通知上游修改
- 避免双向绑定造成的意外更新循环

#### 5.2 不可变数据优先

- 修改数据时创建新副本，不修改原数据
- 减少副作用，便于调试和变更追踪
- 在多线程/并发场景下天然线程安全
- 充分利用语言特性：Rust 的 `&self` vs `&mut self`、Java 的 `record`、Python 的 `dataclass(frozen=True)`

#### 5.3 状态归属原则

- **状态放在离使用者最近的地方**：只被一个模块使用的状态不下放到共享层
- **最小共享范围**：需要共享时，只提升到需要共享的最近公共父级
- **派生状态不要存**：能计算出来的值不存为状态，避免不一致

### 六、可测试性设计

#### 6.1 依赖注入

- 不在构造器或函数内部创建依赖对象，从外部传入
- 方便测试时替换为 mock 或 stub
- 各种语言实现方式：构造函数参数、接口参数、函数参数默认值、DI 容器

#### 6.2 副作用隔离

- 纯逻辑（计算、判断、转换）与 IO 操作（网络、文件、数据库）分离到不同函数
- 纯函数可独立单元测试；IO 函数用集成测试覆盖

```
            ┌──────────────────────┐
  输入 ──→  │  纯逻辑层（无副作用）  │ ──→ 输出
            │  可独立单元测试       │
            └──────────┬───────────┘
                       │
            ┌──────────▼───────────┐
            │  IO 层（有副作用）     │ ──→ 外部系统
            │  用集成测试覆盖       │
            └──────────────────────┘
```

#### 6.3 测试边界清晰

| 测试类型 | 范围 | 测试内容 |
|----------|------|----------|
| **单元测试** | 单个函数/模块 | 纯逻辑：计算、判断、转换 |
| **集成测试** | 模块间交互 | 接口契约、数据流贯通 |
| **端到端测试** | 全系统 | 用户操作流程 |

## 架构健康检查清单

每完成一个模块或阶段时，对照自查：

| 检查项 | 说明 |
|--------|------|
| 模块职责清晰 | 每个模块的边界明确，没有职责重叠或缺失 |
| 接口稳定 | 对外暴露的接口在后续不需要大改 |
| 接口深度 | 模块接口小而实现深，不是接口大而实现薄（参考 1.4 深度模块设计） |
| 依赖方向正确 | 高层模块不依赖低层实现细节，都依赖抽象 |
| 错误有处理 | 异常路径有明确的处理逻辑，不吞异常 |
| 可测试 | 核心逻辑不依赖全局状态，可单独测试，接口即测试面 |
| 无过度设计 | 没有为尚未出现的需求预留复杂性，没有单一适配器的假想缝 |
| 副作用隔离 | IO 操作与纯逻辑分离 |

## 禁止行为

- ❌ 没有接口设计就直接写实现
- ❌ 为了"更快完成"而牺牲架构清晰度
- ❌ 深层继承链（>3 层）或用继承复用行为
- ❌ 链式调用穿过 3 个以上对象（违反最少知识原则）
- ❌ 第一次出现重复就急着重构抽象
- ❌ 吞异常不传播上下文
- ❌ 混放业务异常和技术异常
- ❌ 状态放在共享层但只被一个模块使用
