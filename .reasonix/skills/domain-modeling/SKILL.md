# 领域建模与共享语言

在设计和编码过程中主动建立和打磨项目的领域模型。核心理念来自 Eric Evans《领域驱动设计》：**统一语言（Ubiquitous Language）**——让开发者、AI、代码、文档使用同一套术语。

## 为什么需要这个

| 问题 | 后果 | 解决 |
|------|------|------|
| AI 每次都要重新理解项目术语 | 啰嗦、误解、命名不一致 | `CONTEXT.md` 一次性定义 |
| 同样的架构问题被反复争论 | 浪费时间 | ADR 记录决策，一劳永逸 |
| 代码中的名字和对话中的名字不一致 | 找不到代码、理解困难 | 统一语言贯穿所有层次 |

## 文件结构

大多数项目只有一个上下文：

```
/
├── CONTEXT.md              ← 项目术语表（唯一真相来源）
├── docs/
│   └── adr/                ← 架构决策记录
│       ├── 0001-xxx.md
│       └── 0002-xxx.md
└── src/
```

如果项目很大，可以有多个子上下文：

```
/
├── CONTEXT-MAP.md           ← 列出各子上下文的位置和关系
├── docs/adr/                ← 跨上下文的系统级决策
└── src/
    ├── ordering/
    │   ├── CONTEXT.md
    │   └── docs/adr/
    └── billing/
        ├── CONTEXT.md
        └── docs/adr/
```

**懒创建**：只在有内容要写时才创建文件。没有 `CONTEXT.md`？等第一个术语确定后再创建。没有 `docs/adr/`？等第一个需要记录的决策出现后再创建。

## CONTEXT.md 格式

```markdown
# {上下文名称}

{一两句话描述这个上下文是什么，为什么存在}

## Language

**术语1**：
一句话定义它是什么。
_Avoid_: 替代词1, 替代词2

**术语2**：
一句话定义它是什么。
_Avoid_: 替代词
```

### 规则

1. **要有观点**：当多个词可以表达同一概念时，选最好的那个，其余列在 `_Avoid_` 下
2. **定义要紧凑**：最多一两句话。定义它**是什么**，不是它做什么
3. **只包含项目特定的术语**：通用编程概念（超时、错误类型、工具模式）不属于这里
4. **按自然群组组织**：相关术语放在同一个小标题下
5. **不含实现细节**：CONTEXT.md 是术语表，不是规格说明、草稿本或实现决策库

## ADR（架构决策记录）格式

```
docs/adr/0001-slug.md
docs/adr/0002-slug.md
```

### 模板

```markdown
# {决策简短标题}

{1-3 句话：什么上下文，做了什么决定，为什么。}
```

就这么简单。一个 ADR 可以只有一段话。价值在于记录了**做了决定**和**为什么**，而不是填满所有章节。

### 何时创建 ADR

三个条件**全部**满足时才创建：

1. **难以逆转**：以后改主意的成本很高
2. **没有上下文会很惊讶**：未来的读者看到代码会想"他们为什么这样做？"
3. **真正的权衡结果**：确实有替代方案，你基于特定理由选了一个

如果决定很容易逆转，跳过——你随时可以改。如果不令人惊讶，没人会好奇。如果没有真正的替代方案，没什么可记录的。

### ADR 适合记录什么

- **架构形态**："使用 monorepo"、"写模型用事件溯源，读模型投影到 Postgres"
- **上下文间集成模式**："Ordering 和 Billing 通过领域事件通信，不是同步 HTTP"
- **有锁定效应的技术选型**：数据库、消息总线、认证提供商、部署目标
- **边界和范围决策**："Customer 数据由 Customer 上下文拥有，其他上下文只通过 ID 引用"
- **刻意偏离常规路径**："用原生 SQL 而不是 ORM，因为 X"
- **代码中不可见的约束**："不能用 AWS 因为合规要求"

## 会话中的行为

### 1. 和术语表对照

当用户使用的术语与 `CONTEXT.md` 中已有语言冲突时，立即指出。例如："你的术语表定义 'cancellation' 为 X，但你似乎指的是 Y——到底是哪一个？"

### 2. 收紧模糊语言

当用户使用模糊或过载的术语时，提出精确的规范术语。例如："你说 'account'——是指 Customer 还是 User？这两个是不同的概念。"

### 3. 用具体场景测试

讨论领域关系时，用具体场景做压力测试。发明边界场景来逼迫用户精确化概念之间的边界。

### 4. 与代码交叉验证

当用户声称某事如何运作时，检查代码是否一致。如果发现矛盾，提出："你的代码取消的是整个 Order，但你刚才说部分取消是可以的——哪个是对的？"

### 5. 即时更新 CONTEXT.md

术语一旦确定，立即更新 `CONTEXT.md`。不要攒着批量更新——在它发生时马上记录。

### 6. 审慎提供 ADR

只在三个条件全部满足时才提议创建 ADR。如果用户拒绝某个候选方案且理由有分量，可以提议记录为 ADR。例如："要我把它记录为 ADR 吗？这样以后的架构审查就不会重新建议这个方案了。"

## 禁止行为

- ❌ 把 CONTEXT.md 当需求文档或设计文档写
- ❌ 把通用编程术语放进 CONTEXT.md
- ❌ 为显而易见的决定创建 ADR（"我们用 React 因为是前端项目"）
- ❌ 攒着多个术语一起更新 CONTEXT.md
- ❌ 术语表有冲突时不指出
