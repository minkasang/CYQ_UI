# 开源项目法律追踪

> **给AI的话：** 在引入任何第三方代码、库、素材、公式、算法到本项目之前，必须先读此文件并查清其 License。
> 违反开源协议 = 违法风险，可能导致项目被迫开源、赔偿、被起诉。
> **本文件与 0-启动入口.md 同等优先级**，必须在写代码前先读完。

---

## ⚠️⚠️⚠️ 最高优先级：禁止复制 GPL 代码的硬规定 ⚠️⚠️⚠️

### 🛑 红线（绝对不能做）

```
❌ 绝对禁止直接复制 GPL/AGPL 协议的源代码到自己项目
❌ 绝对禁止 "稍微改一下" GPL 代码再贴过来
❌ 绝对禁止 把 GPL 文件 include 到自己代码里
❌ 绝对禁止 把 GPL 文件作为自己项目的子模块链接
❌ 绝对禁止 AI 在没问用户的情况下，复制 GPL 项目的任何源码
```

**违规后果：整个项目被迫开源（GPL-3.0），商业发布受限。**

### ✅ 完全允许的做法（"参考重写"是合法的）

```
✅ 阅读 GPL 项目源码，理解它的算法思想
✅ 看完之后把思路忘掉，自己从零写一个同样功能的模块
✅ 在自己的代码注释里写："算法思想参考自 XXX 项目，协议为 GPL-3.0，本实现为 Clean Room 重写"
✅ 复制算法的"思想"（不受版权保护）但不复制"具体表达"（受版权保护）
✅ 学习设计模式、架构思路（这些本身不受版权保护）
```

### 📌 关键法律概念：Clean Room（净室实现）

- **允许**：一个人读 GPL 源码 → 用自己的话描述给另一个人 → 第三个人根据描述重写代码
- **关键**：写代码的人**不能看**原 GPL 代码
- **这是合法的**——算法/思想不受版权保护，只有"具体代码表达"受保护

### 一句话总结

> **可以看，可以学，可以自己写，但**不能**Ctrl+C/V GPL 项目的代码。**

## 一、为什么必须有这个文件

1. **法律义务**：每种开源协议（MIT、GPL、Apache、CC-BY 等）有不同义务，混用时互相冲突会污染整个项目
2. **追溯链条**：将来项目要发布到任何渠道时，可能需要列出所有依赖和它们的 License
3. **传染性陷阱**：GPL、AGPL 这种 Copyleft 协议有"病毒式传染"——一旦引入，整个项目必须开源
4. **AI 失忆防护**：新 AI 接手时如果不知道有 GPL 项目，会毫无防备地复制代码，事后才发现已踩雷

## 三、🔴 红线警告：GPL/AGPL 风险

### 什么是 GPL-3.0

GPL-3.0 是 **强 Copyleft（著佐权）** 协议。它的核心条款：

> 如果你把 GPL-3.0 的代码"合并"到你的项目中，**你的整个项目也必须以 GPL-3.0 发布**，
> 且你必须向所有获得你项目的人提供完整的源代码。

### 对本项目的具体影响

| 场景 | 是否允许 | 后果 |
|------|----------|------|
| 看完 GPL 源码后**自己重写**算法 | ✅ 允许 | Clean Room 实现，不传染 |
| **直接复制** GPL 的源码文件到自己项目 | ❌ 禁止 | 整个项目被迫 GPL-3.0 |
| 修改 GPL 的代码后集成 | ❌ 禁止 | 衍生作品必须 GPL-3.0 |
| **仅阅读** 它的算法思想，不复制代码 | ✅ 允许 | 思想不受版权保护 |

## 四、引用/复用开源项目的安全做法

### ✅ 安全（任意 License 都允许）
1. **阅读并理解**算法思想 → 用自己的代码重写
2. 引用论文/文档文字 → 注明出处
3. 调用公开的 REST API（不复制服务端代码）
4. 学习设计模式 → 模式本身不受版权保护
5. 复制 License 头部的版权声明到 `THIRD_PARTY_LICENSES.txt`

### ⚠️ 危险（必须按协议做）
1. MIT / Apache / BSD：保留版权声明 + License 副本
2. CC-BY：注明作者和来源
3. CC-BY-SA：同上 + 你的衍生作品也用 CC-BY-SA
4. MPL-2.0：修改过的文件必须开源，但其他文件可以闭源

### 🔴 禁止（除非项目整体开源）
1. **GPL / LGPL / AGPL**：直接复制代码 = 整个项目必须开源
2. **SSPL**：连服务端都受传染，禁止商用
3. **Unlicense / Public Domain**：可商用但需保留声明

## 五、AI 行为约束（强制）

### 5.1 引入新依赖前必须做的 3 件事

1. 打开该项目的 `LICENSE` / `LICENSE.md` / `LICENSE.txt` 文件，**完整读完**（不是只看 README 里的 "License is XXX" 一句话）
2. 把 License 类型记录到本文件 §二 的表格
3. 在 0-启动入口.md 或 `AI开发规范/任务拆分/共享任务清单.json` 中告知用户"我准备引入 XXX 协议的项目"

### 5.2 复制代码前的 3 个确认

1. 协议是否允许商业用途？
2. 协议是否传染（Copyleft）？
3. 我是否能在自己项目中加 License 声明？

任意一个答案为"否/不确定"→ **停下来问用户，不要自作主张**

### 5.3 引用算法/公式的规范

即使不复制代码，引用开源项目的算法/公式也应在代码注释中写：
```typescript
// 算法参考：XXX 项目的 YYY 公式（MIT 协议）
// 链接：https://github.com/xxx/yyy
// 本实现为独立重写
```
