# Blueprint 框架 — 跨项目迁移指南

> Blueprint（蓝图框架）是一套AI协作开发治理工具。
> 本指南说明如何将本框架从一个项目迁移到另一个项目。

---

## 框架结构

```
AI开发规范/              ← 核心治理层（大部分可直接拷贝）
├── 0-启动入口.md         ← 新AI入职第一份（需修改项目简介部分）
├── 1-行为纪律.md         ← [通用] AI行为红线
├── 2-编码规范.md         ← [通用但有语言特定] 需要根据新项目的语言/引擎微调
├── 3-开发流程.md         ← [通用] 7阶段（含阶段0资源审计）
├── 4-需求挖掘流程.md     ← [通用] 选择题式需求探索
├── 5-代码审查清单.md     ← [通用] 33项代码审查+4项框架自审计（8大类）
├── 6-修订记录.md         ← [项目特定] 每个项目的独立修订历史
├── 7-Git管理策略.md      ← [通用] Git分支/Commit/Tag规范
├── 8-开源项目法律追踪.md ← [通用] 开源协议追踪（GPL传染防护）
├── 任务拆分/             ← [通用] 大需求逐步骤施工清单
│   ├── 共享任务清单.md   ← [项目特定] AI和用户各自的待办
│   └── _模板_任务拆分.md ← [通用] 模板
├── 资源审计报告.md       ← [项目特定] 开源项目盘点
└── 框架迁移指南.md       ← 本文件

产品设计/                 ← [项目特定] 游戏/产品设计文档
参考源码/                 ← [项目特定] 第三方源码（低风险，MIT/BSD 等）
需求存档/                 ← [项目特定] 需求确认文档
```

> **关于高风险开源项目（GPL/AGPL）的处理**：
> - 不要把 GPL/AGPL 等强传染开源项目源码放进主项目的 `参考源码/`
> - 建议剥离到**独立拆解项目**，主项目只接收"脱敏后的算法规范"作为输入
> - 详见 [8-开源项目法律追踪.md](8-开源项目法律追踪.md) §三 红线警告

## 迁移步骤（5分钟搞定）

### 步骤1：复制治理层

```
将 旧项目/AI开发规范/  整个文件夹复制到 新项目/AI开发规范/
将 旧项目/需求存档/     复制到 新项目/需求存档/（空文件夹）
```

### 步骤2：修改 `0-启动入口.md`

打开 `新项目/AI开发规范/0-启动入口.md`，修改以下内容：
- 第1节「项目是什么」→ 写上新项目的简介
- 第4节「项目当前进度」→ 重置为初始状态
- 第5节「技术栈」→ 写上新项目的技术栈
- 第8节「目录结构」→ 更新为实际目录

### 步骤3：根据新项目微调 `2-编码规范.md`

如果新项目用不同语言/引擎：
- Python → 改命名规则为 snake_case
- JS/TS → 改命名规则为 camelCase
- 非Unity → 删除Unity特定规范章节

### 步骤4：清空 `6-修订记录.md`

删掉旧的修订条目，保留模板开头。

### 步骤5（可选）：创建 `产品设计/` 和 `参考源码/`

按新项目的实际需求创建。

## 新项目启动时的AI指令

只需对AI说：

> "先读 `AI开发规范/0-启动入口.md`，然后严格执行阶段0：项目资源审计。"

一句话，AI就会按规范行事。
