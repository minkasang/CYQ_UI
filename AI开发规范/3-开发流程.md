# 开发流程步骤

> 本文档定义从"用户提出想法"到"代码落地运行"的完整链路。
> 每一步都必须执行，不可跳步。
> **核心原则：拆小步、跑通一步再下一步、自己测完再交付。**

---

## 整体流程总览

```
用户提出想法
    ↓
[阶段0] 项目资源审计
    ↓
[阶段1] 需求挖掘与澄清    ← 参见《4-需求挖掘流程.md》
    ↓
[阶段2] 需求确认文档（含测试用例）
    ↓ (用户确认)
[阶段3] 技术方案设计
    ↓
[阶段3.5] 任务拆分（强制产出拆分文档）
    ↓
[阶段4] 渐进式代码实现（每步实现 → 运行验证 → 通过后再下一步）
    ↓
[阶段4.5] 联调（多模块集成验证）
    ↓
[阶段5] 自我审查（代码审查 + 运行测试）
    ↓
[阶段6] 交付（附带测试通过证据 + 已知问题清单）
    ↓
[阶段7] 用户验收
    ↓ (有问题则回到对应阶段修正)
完成
```

---

## 阶段0：项目资源审计（不可跳过）

**时机：** 首次进入项目时，或用户提到新的参考项目/GitHub仓库时。

**输入：** 项目目录结构与用户口头提及的资源
**输出：** 资源审计报告

**执行步骤：**
1. 从所有设计文档中提取提到的参考项目/资源清单
2. 逐一验证每个资源是否物理存在于磁盘上
3. 存在的 → 分析其核心价值、可复用模块、局限性
4. 缺失的 → **立刻告知用户**，询问是否需要下载，或AI自己搜索替代方案提供链接
5. 输出审计报告，记录到 `AI开发规范/6-修订记录.md`

**审计报告格式：**
```
=== 资源审计报告 ===
日期：2026-xx-xx

| 项目/依赖 | 状态 | 核心价值 | 可复用模块 | 局限 |
|-----------|------|---------|-----------|------|
| liquid-glass-react | ✅已发布npm | 流体玻璃React组件 | LiquidGlass / GlassPanel | WebGL兼容性 |
| react-markdown | ✅已安装 | Markdown渲染引擎 | Markdown内容渲染与预览 | 需配置remark-gfm |
| zustand | ✅已安装 | 轻量状态管理 | 全局Store | 无持久化中间件（已自封装） |
| date-fns | ✅已安装 | 日期处理 | 日期计算与格式化 | — |

建议操作：
- 如需新增依赖，先查 License 再安装
- AI自行搜索到的其他推荐库：xxx
```

**此阶段不写任何代码，不清点完不开工。**

## 阶段2：需求确认文档（含测试用例）

**输入**：阶段1的输出
**输出**：书面需求确认文档（存放在 `需求存档/` 下）

文档必须包含：
```markdown
# 需求确认：xxx功能

## 1. 功能描述
（用AI自己的话复述）

## 2. 核心交互流程
步骤1 → 步骤2 → 步骤3

## 3. 输入输出定义
- 触发条件：xxx
- 输入数据：xxx
- 输出结果：xxx
- 失败处理：xxx

## 4. 边界条件处理
| 边界场景 | 处理方案 |
|----------|----------|
| xxx      | yyy      |

## 5. 排除范围
本需求不做以下内容：
- xxx

## 6. 优先级
- P0（必须）：xxx
- P1（重要）：xxx
- P2（锦上添花）：xxx

## 7. 影响范围
- 需要新建的文件：xxx
- 需要修改的现有文件：xxx
- 可能受影响的模块：xxx

## 8. 测试用例 / 验收标准（新增）
| # | 测试场景 | 输入 | 预期输出 | 验证方式 |
|---|----------|------|----------|----------|
| 1 | 正常流程 | xxx | yyy | 手动操作/自动脚本 |
| 2 | 边界条件A | xxx | yyy | 手动操作/自动脚本 |
| 3 | 错误处理 | xxx | 提示"zzz" | 手动操作/自动脚本 |

---
> 用户确认：□ 已确认，可以开始
```

**必须等用户确认后才能进入下一阶段。**

## 阶段3：技术方案设计

**输入**：已确认的需求文档
**输出**：技术实现方案

1. 确定需要新建/修改的文件
2. 确定每个文件的职责（单一职责原则）
3. 确定文件之间的依赖关系
4. 确定数据流向
5. 评估性能影响

输出示例（简短即可）：
```
- 新建 src/store/useXxxStore.ts：负责某业务领域的状态管理
  - 依赖 useYyyStore（读取关联数据）
  - 被 XxxPanel 组件订阅
- 新建 src/components/xxx/XxxPanel.tsx：业务界面
  - 依赖 useXxxStore
- 新建 src/types/xxx.ts：业务领域类型定义
```

## 阶段3.5：任务拆分（强制）

**输入**：技术方案
**输出**：`AI开发规范/任务拆分/[需求名]_任务拆分.md`

**规则：**
1. **必须产出拆分文档** —— 没有拆分文档，不能进入代码实现阶段
2. 复制 `_模板_任务拆分.md` 创建新文件，按格式填写
3. 每个步骤必须拆到 **一次对话可完成** 的粒度
4. 每个步骤必须有 **明确的验证方式**（能跑什么、看到什么输出）
5. 标注步骤之间的依赖关系

**拆分粒度检查：**
- 如果看完一个步骤犹豫"这一步做不完吧？"——立刻再拆一层
- 如果一个步骤涉及新建3个以上文件——拆成多个步骤
- 如果一个步骤既改逻辑又改样式——拆成"先跑通逻辑，再调样式"两步

**拆分完成自检（开始写代码前必须逐项打勾）：**
- [ ] 所有步骤都已拆到可一次对话完成的粒度
- [ ] 每个步骤都有明确的验证方式和预期输出
- [ ] 依赖关系已标注，无循环依赖
- [ ] 共享任务清单已同步更新

---

## 阶段4：渐进式代码实现

**核心规则：跑通一步，再下一步。**

1. **按拆分文档的顺序执行**，一次只做一步
2. **先读相关文件，再写代码**
3. **每完成一步，立即运行验证**：
   - 前端项目：`npm run dev` 看效果，或 `npm run build` 检查编译
   - 手动操作验证功能是否符合预期
   - 浏览器控制台检查有无报错
4. **验证通过后才能进入下一步**
5. **验证不通过时，不允许跳过** —— 必须先修复当前步骤
6. 每完成一步，在拆分文档中更新状态（⏳ → ✅），记录到Todo
7. 必须遵守《2-编码规范.md》

**渐进式交付的好处：**
- 用户可以在中间步骤看到半成品，及时纠正方向
- 出问题时的回滚范围小，只影响当前步骤
- 每一步都有明确的"完成"标准，不会无限蔓延

---

## 阶段4.5：联调（多模块集成验证）

**时机：** 拆分文档中所有步骤都已完成（状态全部为 ✅），但尚未进入最终审查前。

**核心目的：** 验证多个新模块之间、以及新模块与已有模块之间的配合是否正常。单步验证只保证单个模块能跑，联调保证**模块组合在一起能跑通**。

**联调检查清单：**

| # | 检查项 | 验证方式 | 通过标准 |
|---|--------|----------|----------|
| 1 | 组件间数据流 | 在界面操作触发数据变化，观察相关组件是否正确响应 | 数据变化后，所有依赖该数据的组件/视图同步更新 |
| 2 | 路由与状态 | 在不同页面间跳转，观察状态是否保持或正确重置 | 跳转后目标页面加载正确，无空白/报错 |
| 3 | Store联动 | 一个Store的action触发后，检查其他订阅该Store的组件 | 订阅方正确收到更新，无重复渲染/遗漏 |
| 4 | 第三方服务联调 | 调用外部API（如AI接口），观察请求-响应链路 | 请求发出 → 正确收到响应 → 响应数据正确渲染到UI |
| 5 | 端到端流程 | 模拟用户完成一个完整业务流程 | 从入口到出口全程无断点，中间状态正确 |
| 6 | 边界穿越 | 在多模块交界处测试边界条件 | 模块A的异常输出不会导致模块B崩溃 |

**联调记录格式：**
```
=== 联调记录 ===
日期：2026-xx-xx

| # | 检查项 | 结果 | 问题描述 |
|---|--------|------|----------|
| 1 | 组件间数据流 | ✅通过 | — |
| 2 | 路由与状态 | ⚠️问题 | 从A页跳B页后，B页store数据未重置 |

修复后重新联调：□ 已完成
```

**联调不通过怎么办：**
1. 定位是哪个模块之间的配合出了问题
2. 回到阶段4修复对应模块（或调整模块接口）
3. 修复后重新执行阶段4的该步骤验证 + 阶段4.5全部联调项

---

## 阶段5：自我审查（代码审查 + 运行测试）

**输入**：全部代码
**输出**：审查报告 + 测试运行记录

本阶段分两个并行环节，**都必须完成**：

### 5.1 代码审查
对照《5-代码审查清单.md》逐项检查，输出审查报告。

### 5.2 运行测试（新增，强制）

**必须执行的测试：**

| # | 测试项 | 执行方式 | 通过标准 |
|---|--------|----------|----------|
| 1 | 编译检查 | `npm run build` | 无编译错误 |
| 2 | 开发服务器启动 | `npm run dev` | 能正常启动，控制台无报错 |
| 3 | 需求测试用例逐个执行 | 手动操作 | 输入X → 输出符合预期 |
| 4 | 边界条件测试 | 手动操作 | 边界场景处理正确 |
| 5 | 现有功能回归 | 手动操作核心流程 | 已有功能未被破坏 |

**测试记录格式：**
```
=== 测试运行记录 ===
日期：2026-xx-xx

| # | 测试场景 | 结果 | 证据 |
|---|----------|------|------|
| 1 | 编译检查 | ✅通过 | 无报错，build成功 |
| 2 | 用例1：正常流程 | ✅通过 | 截图/描述 |
| 3 | 用例2：边界条件 | ⚠️问题 | 描述问题，已修复/待修复 |

结论：可以交付 / 需修复后重测
```

**如果测试不通过：**
1. 记录失败场景和现象
2. 修复问题
3. **从头重新执行全部测试**（不能只测刚修的部分）
4. 更新测试记录

---

## 阶段6：交付

**输入**：审查报告 + 测试运行记录
**输出**：交付说明

AI向用户输出交付说明，必须包含：

```
=== 功能交付：xxx ===

1. 做了什么（一句话总结）
2. 改了哪些文件（列出新建和修改的文件）
3. 测试通过证据（附上测试运行记录）
4. 如何验证（用户自己可以操作验证的步骤）
5. 已知问题（如果有，诚实列出，不要隐瞒）
6. 后续可优化方向（可选，主动补充）
```

**交付标准（必须全部满足）：**
- [ ] 代码审查全部通过
- [ ] 联调检查全部通过（阶段4.5）
- [ ] 所有测试用例执行通过
- [ ] `npm run build` 编译成功
- [ ] 已有核心功能未被破坏（回归测试通过）
- [ ] 拆分文档中所有步骤状态为 ✅

**不满足交付标准 = 不能交付。** 继续修复，直到满足。

---

## 阶段7：用户验收

等用户反馈。用户可能：
- ✅ 满意通过 → 更新任务拆分状态为全部完成，归档需求文档
- ⚠️ 提出小问题 → 走Bug修复流程
- ❌ 方向偏差 → 回到阶段1重新澄清需求

---

## 补充：Bug修复流程

当用户报告Bug时，使用以下流程：
1. **复述Bug现象** —— 确认理解正确
2. **写最小复现步骤** —— 记录到Bug修复记录
3. **定位问题代码**
4. **分析根因**
5. **提出修复方案** —— 用户确认后修复
6. **修复后执行回归测试**（新增）：
   - 验证Bug已修复
   - 验证修复没有破坏其他功能（执行核心流程回归）
   - 记录到测试运行记录
7. **自审查**
8. **重新交付**
