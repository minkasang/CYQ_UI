## 二、TypeScript / React 编码规范

### 2.1 命名规则

| 类型 | 规则 | 示例 |
|------|------|------|
| 组件名 | PascalCase | `UserProfile`、`DataTable` |
| 接口/类型 | PascalCase | `UserData`、`ApiConfig` |
| 枚举值 | PascalCase | `ThemeMode.Dark` |
| 函数/方法 | camelCase | `handleSubmit()` |
| 私有函数 | _camelCase（可选） | `_validateInput()` |
| 常量 | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE` |
| 局部变量 | camelCase | `currentItem` |
| 布尔变量 | is/has/can 前缀 | `isLoading`、`hasError` |
| Hook | use 前缀 + PascalCase | `useUserStore` |
| Zustand Store | use + 领域 + Store | `useSettingsStore` |

### 2.2 文件组织

- 一个文件只放一个**主组件**（或紧密相关的几个小组件）
- 文件名与主导出组件名一致：`UserProfile.tsx`
- 工具函数、类型定义可集中放在 `utils/`、`types/` 目录
- Store 文件按领域拆分：`useUserStore.ts`、`useSettingsStore.ts`

### 2.3 注释规范

```typescript
/**
 * 根据条件筛选数据列表
 * @param items 原始数据列表
 * @param filter 过滤条件
 * @returns 过滤后的列表
 */
function filterItems(items: Item[], filter: FilterType): Item[] { ... }
```

简单逻辑用单行注释：
```typescript
// 本地优先：先写 localStorage，再触发状态更新
```

### 2.4 空值/边界检查

所有接收外部输入的方法必须做空值检查：
```typescript
function processItem(item: Item | undefined) {
  if (!item) {
    console.error('[ProcessItem] item 为 undefined，跳过处理。');
    return;
  }
  // ...
}
```

### 2.5 魔法数字禁令

代码中不允许出现未命名的数字常量：
```typescript
// 错误
if (items.length < 10) { ... }

// 正确
const MAX_PAGE_SIZE = 10;
if (items.length < MAX_PAGE_SIZE) { ... }
```

## 三、React / Vite 特定规范

### 3.1 组件编写

- 优先使用**函数组件** + **Hooks**
- Props 必须定义 TypeScript 接口
- 避免在 JSX 中写复杂内联逻辑，抽成变量或函数

```typescript
interface DataRowProps {
  data: DataItem;
  onSelect: (id: string) => void;
}

export function DataRow({ data, onSelect }: DataRowProps) {
  const isStale = useMemo(() => checkStale(data.updatedAt), [data.updatedAt]);

  return (
    <div className={isStale ? 'text-red-500' : ''}>
      {/* ... */}
    </div>
  );
}
```

### 3.2 性能红线

- 不在 `useEffect` 中做无依赖的频繁重计算
- 大数据列表必须做虚拟滚动或分页
- 事件处理器用 `useCallback` 缓存（仅当需要时，不要滥用）
- 避免在 render 中创建新对象/数组（导致子组件不必要的重渲染）

### 3.3 状态管理

- 局部状态用 `useState`
- 跨组件共享状态用 Zustand Store
- 禁止用 `useState` 存储大量结构化数据（数据量增长到一定规模时应做分页/索引或移出组件状态）

### 3.4 样式规范

- 使用 Tailwind CSS 工具类
- 自定义主题色在样式配置文件（如 `tailwind.config.js`）中配置
- 全局样式在 `src/index.css` 中定义
- 组件级特殊样式可用内联 style 或 CSS Module（本项目统一用 Tailwind）

## 四、规范一致性保证

### 4.1 规范即代码

规范文件使用 Git 管理，修改规范必须有明确的 commit message。

### 4.2 规范文档交叉引用标准

> 规范不是孤岛。改一个可能牵一发而动全身。必须建立显式的交叉引用机制。

**每个规范文件必须包含以下三部分：**

**1. 头部关联规范列表（文件最顶部）**
```markdown
> 关联规范：[启动入口](0-启动入口.md)、[行为纪律](1-行为纪律.md)、[开发流程](3-开发流程.md)
```

**2. 文末"相关规范索引"章节**
```markdown
---

## 相关规范索引

| 相关规范 | 关系说明 | 链接 |
|----------|---------|------|
| 行为纪律 | 编码规范需配合行为纪律执行 | [链接](1-行为纪律.md) |
| 代码审查清单 | 审查时对照本规范检查 | [链接](5-代码审查清单.md) |
```

**3. 修改规范时的强制流程**
```
修改规范X时：
  Step 1: 读取规范X的"相关规范索引"
  Step 2: 逐一读取所有相关规范
  Step 3: 判断：我的修改会影响它们吗？
    - 如果会 → 同步修改相关规范，并更新它们的"相关规范索引"
    - 如果不会 → 在PR/提交说明中注明"已检查，无影响"
  Step 4: 检查是否有其他规范引用了规范X（全文搜索规范X的文件名）
  Step 5: 那些被引用的规范，它们的"头部关联规范列表"和"相关规范索引"需要更新吗？
```

**禁止行为：**
- ❌ 只改一个规范，不检查相关规范
- ❌ 头部关联规范列表只有名字没有链接
- ❌ 相关规范索引不写"关系说明"或写"相关"
- ❌ 删除了某规范的内容，但没更新引用它的其他规范

### 4.3 规范不可回溯修改

一旦某条规范被应用到代码中，就不能随意修改该规范。
如果必须修改，需要：
1. 列出所有受影响的代码文件
2. 制定迁移计划
3. 逐文件修改并记录
