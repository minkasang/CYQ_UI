# AI 圆桌工作坊 — 功能设计规范 v3.0

> 创建: 2026-06-28  
> 定位: 将“多 AI 群聊”产品化为“专家组分析想法”的工作流  
> 适用对象: 个人工作台的终端用户

---

## 一、产品定位

AI 圆桌工作坊不是普通聊天，也不是让多个 AI 自由闲聊。它是一套受控的专家协作流程：

```
用户提出模糊想法
  → 主持人澄清
  → 多位专家分镜分析
  → 风险官反方审查
  → 主持人收敛
  → 输出结构化方案
```

用户获得的核心价值是：把“我有个想法，但不知道能做成什么”转化为“我清楚知道第一版做什么、为什么做、怎么做、风险在哪”。

---

## 二、功能清单

### 2.1 MVP 必须做

| 功能 | 说明 | 用户价值 |
|------|------|----------|
| 想法输入 | 用户输入一句或一段模糊想法。 | 降低启动成本，不要求用户先写完整需求。 |
| 主持人澄清 | 主持人判断信息是否足够，不足时提出 1-3 个关键问题。 | 防止 AI 在错误前提上发散。 |
| 固定专家组 | 内置主持人、产品专家、体验设计师、技术架构师、风险官、执行规划师。 | 保证分析覆盖关键视角。 |
| 阶段化讨论 | 按澄清、发散、质疑、收敛、产出推进。 | 讨论有秩序，不变成噪音。 |
| 专家分镜发言 | 每个专家只从自己的职责角度发言。 | 让用户看到不同“镜头”的智慧。 |
| 反方审查 | 风险官必须指出不靠谱、过度设计、不可落地之处。 | 抑制 AI 讨好式回答。 |
| 结构化产出 | 输出目标、用户路径、MVP 范围、功能清单、技术建议、风险、下一步。 | 结果可保存、可执行、可继续拆解。 |
| 保存结果 | 工作坊产物可保存在会话中。 | 方便用户回顾和继续追问。 |
| 继续追问 | 用户可对任一专家或最终方案继续提问。 | 支持从圆桌产物进入深入讨论。 |

### 2.2 v3.0 明确不做

| 不做 | 原因 |
|------|------|
| 无限自由群聊 | 容易发散、重复、无结论。 |
| 用户自由配置复杂流程图 | MVP 阶段会增加理解成本。 |
| 让所有 Agent 永久记忆所有内容 | 涉及隐私、上下文污染和性能风险。 |
| 自动创建真实开发任务 | 先输出建议，是否进入开发流程由用户确认。 |
| 取代普通 AI 聊天 | 普通问答仍由 AI 模块承担。 |
| 推翻 1v1 与群聊分离存储 | 已有 ADR 0001，不在本功能中重开。 |

### 2.3 后续版本可做

| 功能 | 触发条件 |
|------|----------|
| 自定义专家模板 | MVP 角色稳定后再开放。 |
| 从每日灵感一键发起圆桌 | 每日灵感和圆桌工作坊边界稳定后。 |
| 圆桌产物转任务拆解 | 工作坊产出格式稳定后。 |
| 专家记忆模块 | 明确隐私与存储策略后。 |
| 工具调用 | 基础工作流稳定且测试覆盖足够后。 |

## 三、用户路径

### 路径 1：从空白想法开始

```
首页 / Agent 模块
  → 点击「AI 圆桌工作坊」
  → 输入想法
  → 主持人澄清
  → 用户补充或跳过
  → 专家组分析
  → 风险官质疑
  → 主持人总结
  → 保存结构化方案
```

### 路径 2：从现有群聊升级为圆桌

```
Agent 群聊
  → 选择「圆桌模式」
  → 选择或使用默认专家组
  → 输入本次议题
  → 按工作坊阶段推进
  → 生成最终方案
```

### 路径 3：继续追问某个专家

```
查看最终方案
  → 点击某条专家观点 / @某个专家
  → 进入追问
  → 该专家回答
  → 主持人可选择更新最终方案
```

### 路径 4：从每日灵感发起（后续）

```
每日灵感
  → 选择一条灵感
  → 点击「交给圆桌分析」
  → 自动带入灵感内容
  → 进入工作坊
```

## 四、默认专家组

| 角色 | 职责 | 发言边界 |
|------|------|----------|
| 主持人 | 控制阶段、澄清问题、合并结论。 | 不抢专家视角，不泛泛鼓励。 |
| 产品专家 | 判断用户、问题、价值、MVP 边界。 | 不写具体代码实现。 |
| 体验设计师 | 设计入口、路径、状态、反馈、视觉体验。 | 不替代产品价值判断。 |
| 技术架构师 | 判断模块边界、数据流、接口、实现复杂度。 | 不提前写代码。 |
| 风险官 | 找失败点、歧义、过度设计、成本风险。 | 必须给出可处理的风险，不只否定。 |
| 执行规划师 | 把方案拆成阶段、任务、验收标准。 | 不绕过用户确认直接进入开发。 |

---

## 五、工作坊阶段

| 阶段 | 用户看到什么 | 系统内部规则 |
|------|--------------|--------------|
| 澄清 | 主持人提出关键问题，最多 3 个。 | 信息明显不足才问；用户可跳过。 |
| 发散 | 专家卡片依次或分组输出观点。 | 每个专家只按职责输出，不互相重复。 |
| 质疑 | 风险官列出关键风险，专家可回应。 | 至少覆盖可行性、范围、体验、实现风险。 |
| 收敛 | 主持人展示共识与取舍。 | 删除重复内容，保留分歧和未决项。 |
| 产出 | 显示结构化方案卡片。 | 产物必须可保存、可追问、可转下一步。 |

## 七、状态覆盖

| 场景 | 表现 | 规则 |
|------|------|------|
| 无 Agent | 显示默认专家组模板，提示需先创建或自动创建预设 Agent。 | 不进入空圆桌。 |
| 用户输入为空 | 开始按钮 disabled，提示“先写下你的想法”。 | 外层输入校验。 |
| 想法过短 | 主持人优先澄清，不直接发散。 | 例如少于一句完整描述。 |
| 用户跳过澄清 | 继续进入发散，但产出标记“基于有限信息”。 | 管理预期。 |
| 某专家 API 失败 | 该专家卡片显示失败，其他专家继续。 | 单点失败不阻塞流程。 |
| 多个专家失败 | 主持人提示“分析不完整”，允许重试失败专家。 | 避免假装完整。 |
| 风险过多 | 主持人分级：必须处理 / 可接受 / 后续观察。 | 防止风险列表淹没用户。 |
| 产出为空 | 显示错误态，允许重新生成总结。 | 不保存空产物。 |
| 用户切换会话 | 当前阶段状态保存；请求中断或忽略过期结果。 | 防止串消息。 |
| 长内容 | 讨论区保留摘要，完整内容可展开。 | 控制上下文和视觉负担。 |

## 八、先进协作机制

### 8.1 独立初稿

发散阶段第一轮必须使用“独立初稿”：

- 产品专家、体验设计师、技术架构师、价值/执行类专家先各自输出第一版判断。
- 第一轮输出前，不让专家看到其他专家的具体观点。
- 主持人只提供用户想法、澄清信息、角色职责和当前阶段目标。
- 第一轮完成后，才进入互评、质疑和收敛。

这样做是为了防止第一个发言的专家形成锚点，导致其他专家跟随附和。

### 8.2 发言候选池

每个阶段只开放必要专家发言：

| 阶段 | 候选池 | 规则 |
|------|--------|------|
| 澄清 | 主持人 | 只问关键问题，不展开分析。 |
| 发散 | 产品专家、体验设计师、技术架构师、执行规划师 | 先独立初稿，再允许补充。 |
| 质疑 | 风险官、被质疑专家 | 风险官先提风险，被点名专家回应。 |
| 收敛 | 主持人、执行规划师 | 合并共识，形成可执行路径。 |
| 产出 | 主持人、质量门禁 | 生成并检查最终结构化方案。 |

### 8.3 质量门禁

最终产物生成后必须通过固定检查：

- 是否说清楚“最终应该做成什么”。
- 是否说清楚“不做什么”。
- 是否有 MVP 范围。
- 是否有用户路径。
- 是否有技术实现建议。
- 是否有风险与取舍。
- 是否有下一步计划。

不通过时，主持人最多重写 2 次；仍不通过则保留草稿并标记“产物未达标”。

### 8.4 用户检查点

工作坊不是 AI 自己跑到底，关键节点要给用户插手机会：

| 检查点 | 触发时机 | 用户可做什么 |
|--------|----------|--------------|
| 方向确认 | 澄清后 | 补充信息、跳过澄清、改变分析重点。 |
| 方向选择 | 发散后 | 选择更感兴趣的方向，让后续收敛围绕它展开。 |
| 产出偏好 | 产出前 | 选择偏产品、偏技术、偏执行或均衡。 |

MVP 中用户检查点可以做成轻量按钮和输入框，不做复杂流程编辑器。

### 8.5 可观测记录

每次圆桌会话至少记录：

- 当前阶段和阶段耗时。
- 每个专家是否发言、失败、跳过。
- 被跳过的原因，如 API Key 缺失、阶段不需要、超时。
- 产物版本和质量门禁结果。
- 本次分析是否完整。

这些记录默认面向系统调试和用户预期管理，不做成复杂日志后台。

## 九、视觉与交互参考

- 延续个人工作台偏好的 macOS 原生感、暗色背景、大片留白、毛玻璃卡片。
- 不做模板感强的“企业后台表格”，更像一个安静的创作室。
- 工作坊主区域使用三栏：专家组、讨论区、阶段产物。
- 阶段进度用轻量进度点，不使用强制跳转。
- 专家卡片只显示必要信息：名称、职责、状态、最近观点。
- 结构化产出使用清晰卡片，不和实时聊天混在一个视觉层级里。

## 十、验收标准

- 用户可以用一句模糊想法启动工作坊。
- 主持人能在信息不足时先问关键问题，而不是盲目生成长文。
- 专家观点能体现不同角色视角，不互相复读。
- 风险官必须提出真实风险，并能转化为取舍建议。
- 最终产出必须包含“做什么、不做什么、MVP、用户路径、实现建议、风险、下一步”。
- 发散阶段第一轮专家观点应相互独立，不被其他专家先发言锚定。
- 最终产物必须经过质量门禁，不合格时重写或标记未达标。
- 至少在澄清后和产出前提供用户检查点。
- 单个专家失败不导致整个工作坊失败。
- 工作坊和普通 AI 聊天、Agent 1v1、智能体群聊的边界清晰。
