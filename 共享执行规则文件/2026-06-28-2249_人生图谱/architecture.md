# 架构设计 - 人生图谱

> 任务名：人生图谱  
> 阶段：MVP 第一阶段  
> 基础模块：`modules/inspiration`

---

## 设计目标

人生图谱不是新建一个孤立大模块，而是在现有 `inspiration` 模块上做领域升级：

- 保持模块 id、路由、持久化 key 不变，保证兼容性。
- 扩展记录模型，让同一条内容可承载碎片、洞察、原则、行动实验。
- UI 默认是漫游模式，降低记录与回看的心理压力。
- 锚定模式只在用户需要改变时出现，避免把系统做成压迫式任务面板。

---

## 质量属性与设计维度

| 质量属性 | 设计措施 |
|----------|----------|
| 可靠性 | 旧数据字段全部可选新增，读取时使用默认值和派生逻辑，避免持久化迁移失败导致页面崩溃。 |
| 可维护性 | 领域类型集中在 `types/index.ts`，状态变更集中在 `useInspirationStore.ts`，UI 只消费小接口。 |
| 可扩展性 | 第一阶段不引入 AI 服务缝；等出现真实 AI 提炼需求时再增加独立服务接口。 |
| 性能效率 | 内在图谱统计通过 `useMemo` 从 items 派生，不写入 store，避免重复状态和同步问题。 |
| 可测试性 | store 新增行为是纯状态变更，可用 Zustand 单元测试覆盖。 |
| 可观测性 | 行动实验保留状态、创建时间、完成时间，可追踪用户是否把洞察转化为行为。 |
| 安全性 | 第一阶段只使用本地持久化，不新增外部网络调用，不上传用户隐私内容。 |
| 兼容性 | `pw-inspiration` persist key、`inspiration` module id、`/inspiration` route 保持不变。 |
| 可移植性 | 仍使用 React + Zustand + Tailwind，不引入平台绑定依赖。 |

## 数据模型

### `InspirationItem` 扩展字段

- `kind`：记录类型，默认 `fragment`。
- `dimensions`：人生维度，如价值观、关系、工作、情绪、身体、学习、创造、原则。
- `insight`：用户或 AI 提炼出的核心洞察。
- `principle`：沉淀出的原则提醒。
- `actionExperiment`：小行动实验，包含标题、触发条件、行动、状态和时间。
- `linkedDiaryIds`：未来与日记联动的预留引用，第一阶段只定义字段不写入。

### 状态归属

- 持久状态：用户输入的原始记录、提炼字段、行动实验状态。
- 派生状态：主题统计、维度统计、最近原则、待行动实验。
- 本地 UI 状态：漫游/锚定模式、搜索、排序、编辑态。

## 模块边界

### `useInspirationStore`

职责：

- 新增记录。
- 编辑记录。
- 维护收藏、触动程度、回顾时间。
- 更新行动实验状态。
- 提供下一条回顾选择。

不负责：

- AI 调用。
- 日记读取。
- UI 布局。
- 主题颜色管理。

### `InspirationSection`

职责：

- 呈现人生图谱 MVP 的主界面。
- 组织漫游模式与锚定模式。
- 从 store 读取记录并派生展示统计。
- 保留现有编辑、删除、搜索和标签输入能力。

不负责：

- 持久化细节。
- 跨模块数据同步。
- AI 提炼算法。

## 与日记系统的关系

第一阶段只保留概念接口：

- `linkedDiaryIds?: string[]` 表示未来可以互相引用。
- 不读取 `useDiaryStore`。
- 不在保存灵感时写日记。
- 后续若要从日记提取洞察，应独立设计 AI 提炼服务与用户确认流程。

---

## 架构自查

- 模块职责清晰：store 管状态，section 管 UI，标签 store 仍只管标签元数据。
- 接口稳定：保留旧方法，新增字段与方法向后兼容。
- 接口深度：不为 AI 预先创建单适配器假缝。
- 依赖方向正确：人生图谱不依赖日记实现。
- 错误有处理：用户空输入不保存，行动实验状态被限制为枚举。
- 可测试：新增 store 行为可直接单元测试。
- 无过度设计：第一阶段只做本地 MVP，不引入服务层。
- 副作用隔离：持久化仍由 Zustand persist 管理，统计派生不产生副作用。
