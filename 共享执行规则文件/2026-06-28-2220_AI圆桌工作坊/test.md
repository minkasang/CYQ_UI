# 测试方案 — AI 圆桌工作坊 v3.0

> 创建: 2026-06-28  
> 当前性质: 测试设计与验收矩阵  
> 说明: 本轮只产出文档，尚未实现代码，因此不记录“已通过”的代码测试结果。

---

## 一、测试目标

AI 圆桌工作坊的测试目标不是验证“AI 说得像不像专家”，而是验证系统是否稳定地把模糊想法推进到可执行方案：

- 阶段流转正确。
- 多 Agent 失败可降级。
- 旧群聊和 1v1 不被破坏。
- 最终产物结构完整。
- 用户能理解当前状态和下一步。

---

## 二、测试边界

| 范围 | 测试类型 | 说明 |
|------|----------|------|
| `phaseMachine` | 单元测试 | 阶段合法流转、非法流转拦截。 |
| `rolePresets` | 单元测试 | 默认专家组完整、职责边界清楚。 |
| `candidateSelector` | 单元测试 | 每个阶段只返回允许发言的专家候选池。 |
| `contextIsolation` | 单元测试 | 独立初稿阶段不泄露其他专家观点。 |
| `artifactBuilder` | 单元测试 | 产物字段完整、空内容失败、版本递增。 |
| `qualityGate` | 单元测试 | 固定 rubric 检查最终产物质量。 |
| `WorkshopOrchestrator` | 单元测试 | mock LLM，验证澄清、发散、质疑、产出。 |
| `groupChatStore` 扩展 | Store 测试 | 旧数据兼容、新工作坊字段持久化。 |
| 工作坊 UI 组件 | 组件测试 | 入口、进度、专家状态、产物卡。 |
| 完整用户路径 | 集成 / 手动 | 从输入想法到保存产物。 |

## 三、测试分类矩阵

| 测试分类 | 场景 | 测试类型 | 验收标准 |
|----------|------|----------|----------|
| 操作间干扰 | 用户启动工作坊后快速切换会话。 | 集成/手动 | 旧请求结果不写入新会话。 |
| 操作间干扰 | 重复点击“开始圆桌”。 | 组件/集成 | 同一会话只创建一个运行中的 runId。 |
| 数据一致性 | 旧群聊无 `mode` 字段。 | Store 单元测试 | 自动视为 `free`，不报错。 |
| 数据一致性 | `mode` 与 `strategy` 同时存在。 | Store/编排单元测试 | `mode` 决定会话类型，`strategy` 不得单独触发工作坊流程。 |
| 数据一致性 | 工作坊产物重新生成。 | Store 单元测试 | `version` 递增，历史版本保留或按设计替换。 |
| 边界与极限 | 用户输入为空。 | 组件测试 | 开始按钮 disabled。 |
| 边界与极限 | 用户输入极短。 | 编排单元测试 | 进入澄清阶段，而非直接产出。 |
| 边界与极限 | 长讨论记录。 | 单元/集成 | 上下文被裁剪或压缩，不无限增长。 |
| 恢复能力 | 单个专家 LLM 超时。 | 编排单元测试 | 该专家失败，其他专家继续。 |
| 恢复能力 | 结构化产物解析失败。 | 编排单元测试 | 保留原始总结，提示可重试。 |
| 用户预期管理 | 用户跳过澄清。 | 集成/手动 | 产物标记“基于有限信息”。 |
| 用户预期管理 | 多个专家失败。 | 集成/手动 | UI 显示“分析不完整”，不假装成功。 |
| 用户预期管理 | 澄清后出现用户检查点。 | 集成/手动 | 用户可补充、跳过或改变分析重点。 |
| 渲染与视觉 | 阶段进度显示。 | 组件测试 | 当前阶段高亮，已完成阶段可识别。 |
| 渲染与视觉 | 最终产物卡片。 | 组件测试 | 七类结果字段清晰展示。 |
| 交互反馈 | 专家卡片处理中。 | 组件测试 | 显示等待/思考状态。 |
| 交互反馈 | @专家追问。 | 集成/手动 | 指定专家响应，不触发全流程重跑。 |
| 交互反馈 | 质量门禁未通过。 | 组件/集成 | 显示重写中或未达标状态，不保存为空产物。 |
| 性能与流畅度 | 6 个专家同时发散。 | 集成/性能观察 | UI 不冻结，状态持续可见。 |
| 性能与流畅度 | 多轮阶段执行。 | 单元/集成 | 最大轮数受控，不无限循环。 |
| 安全与权限 | API Key 缺失。 | 集成/手动 | 不展示 Key 内容，只提示专家不可用。 |
| 安全与权限 | 用户输入包含 prompt 注入。 | 编排测试 | 系统仍按角色职责和阶段规则执行。 |
| 可维护性 | 编排模块删除测试。 | 架构审查 | 删除后复杂度会扩散到 UI，证明模块有价值。 |
| 可维护性 | 规则模块形态检查。 | 代码审查 | `candidateSelector`、`contextIsolation`、`qualityGate` 为纯函数，不引入单实现接口/类。 |
| 可维护性 | 页面组件职责检查。 | 代码审查 | `AgentsChatPage` 不包含完整阶段机和 prompt 拼接。 |
| 可维护性 | 可观测记录完整性。 | 单元/集成 | 每阶段记录发言、失败、跳过、质量门禁结果。 |
| 实际效果 | 防锚定偏差。 | 编排单元测试 | 发散第一轮各专家上下文不包含其他专家初稿。 |
| 实际效果 | 输入一个真实产品想法。 | 手动验收 | 输出包含做什么、不做什么、MVP、路径、技术、风险、下一步。 |

## 四、单元测试设计

### 4.1 `phaseMachine`

| 用例 | 输入 | 期望 |
|------|------|------|
| 从 `clarify` 到 `diverge` | 信息足够 | 允许推进。 |
| 从 `clarify` 直接到 `artifact` | 未经过分析 | 拒绝推进。 |
| 从 `challenge` 到 `converge` | 已记录风险 | 允许推进。 |
| 从 `completed` 回到 `diverge` | 普通追问 | 拒绝，除非显式重新运行。 |

### 4.2 `rolePresets`

| 用例 | 期望 |
|------|------|
| 默认专家组包含主持人、产品、体验、技术、风险、执行。 |
| 每个必选角色都有明确责任。 |
| 风险官不能缺失；缺失时主持人代行并标记不完整。 |
| 角色职责不应全部相同。 |

### 4.3 `artifactBuilder`

| 用例 | 期望 |
|------|------|
| 完整阶段记录生成完整产物。 |
| 空消息不能生成产物。 |
| 缺少风险时产物校验失败或补充“未识别风险”。 |
| 重新生成时版本递增。 |

### 4.4 `candidateSelector`

| 用例 | 期望 |
|------|------|
| `clarify` 阶段只返回主持人。 |
| `diverge` 阶段返回产品、体验、技术、执行等分镜专家。 |
| `challenge` 阶段必须包含风险官。 |
| 风险官缺失时返回主持人代行，并标记不完整。 |
| 候选池为空时阶段不得继续。 |

### 4.5 `qualityGate`

| 用例 | 期望 |
|------|------|
| 缺少 MVP 范围时失败。 |
| 缺少风险与取舍时失败。 |
| 缺少下一步计划时失败。 |
| 完整产物通过并给出合格分数。 |
| 连续失败达到 2 次后停止重写并标记未达标。 |

### 4.6 `contextIsolation`

| 用例 | 期望 |
|------|------|
| 独立初稿阶段不传入其他专家第一轮观点。 |
| 质疑阶段可传入被质疑观点和风险列表。 |
| 收敛阶段可传入所有阶段摘要，但不传入无关长历史。 |

### 4.7 `WorkshopOrchestrator`

| 用例 | mock 行为 | 期望 |
|------|-----------|------|
| 短输入 | 主持人返回澄清问题 | 停在 `clarify`。 |
| 完整输入 | 专家都成功 | 进入产出阶段并生成 artifact。 |
| 独立初稿 | 多专家返回观点 | 第一轮专家上下文互不可见。 |
| 一个专家失败 | 该专家返回 null | 其他专家继续，产物标记部分缺失。 |
| 风险官质疑 | 风险官返回风险列表 | 收敛阶段包含风险取舍。 |
| 质量门禁失败 | 初稿缺少 MVP | 触发重写，最多 2 次。 |
| 用户检查点 | 发散结束 | 状态暂停，等待用户选择方向。 |
| LLM 全部失败 | 全部返回 null | 状态为 `failed` 或 `paused`，不保存空产物。 |

### 4.8 `WorkshopState`

| 用例 | 期望 |
|------|------|
| 创建工作坊状态 | 包含 `topic`、`phase`、`roles`、`artifacts`、`trace`、`status`。 |
| 创建用户检查点 | `checkpoint` 存在且 `status` 为 `paused`。 |
| 质量门禁完成 | `quality` 写入最近一次检查结果。 |
| 专家缺失或失败 | `incompleteReason` 可记录不完整原因。 |
| trace 写入 | 不包含 API Key、完整 prompt 或敏感内容。 |

## 五、集成测试设计

| 路径 | 步骤 | 验收 |
|------|------|------|
| 创建工作坊 | 输入想法 → 选择快速圆桌 → 创建会话。 | `groupChatStore` 中出现 `mode: 'workshop'`。 |
| 阶段推进 | 澄清 → 发散 → 质疑 → 收敛 → 产出。 | 阶段状态按顺序变化。 |
| 旧群聊兼容 | 加载旧 `group_chats.json` 样本。 | 旧群聊仍能打开和发送消息。 |
| 1v1 不受影响 | 打开 Agent 1v1 对话并发送消息。 | 仍使用 `agentChatStore`，不读取工作坊字段。 |
| 失败降级 | 模拟某专家无 API Key。 | 其他专家继续，UI 显示该专家不可用。 |

## 六、手动验收脚本

### 6.1 标准路径

1. 打开 Agent 模块。
2. 进入“AI 圆桌工作坊”。
3. 输入：“我想做一个多 AI 镜头互动功能，让不同专家帮我把想法分析透。”
4. 选择“快速圆桌”。
5. 等待主持人澄清，选择补充或跳过。
6. 查看专家组发散观点。
7. 查看风险官质疑。
8. 查看最终结构化产物。
9. 保存产物。
10. @技术架构师追问实现方式。

验收：

- 阶段可见。
- 专家职责清楚。
- 结果不是普通聊天摘要。
- 最终产物包含完整七类内容。

### 6.2 失败路径

1. 关闭某个专家使用的 API Key。
2. 启动圆桌。
3. 观察该专家失败时其他专家是否继续。
4. 查看最终产物是否标记“不完整”。

验收：

- 不崩溃。
- 不吞失败。
- 不假装所有专家都完成。

### 6.3 兼容路径

1. 打开旧普通群聊。
2. 发送一条消息。
3. 打开 1v1 Agent 对话。
4. 发送一条消息。
5. 再回到工作坊会话。

验收：

- 三种模式互不污染。
- 消息不会串到错误会话。
