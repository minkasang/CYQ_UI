# .trae/rules 与 .trae/skill 对比分析与优化方案

## 一、现状分析

### 1.1 文件定位

#### `.trae/rules/project_rules.md`
- **定位**：项目级执行规则，强制约束
- **目标**：确保 AI 执行任务时遵循规范流程
- **特点**：强调产出文件、流程控制、质量保障

#### `.trae/skill/healthy-architecture/SKILL.md`
- **定位**：架构设计规范与最佳实践
- **目标**：指导架构设计决策
- **特点**：语言无关、理论性强、通用性高

### 1.2 内容对比

| 维度 | rules/project_rules.md | skill/healthy-architecture/SKILL.md | 冲突点 |
|------|----------------------|-------------------------------------|--------|
| **架构原则** | 8个质量属性（可靠性、可维护性、可扩展性、性能效率、可测试性、可观测性、安全性、兼容性、可移植性） | 6大设计维度（设计原则、项目结构、接口与契约、错误处理、数据流与状态管理、可测试性设计） | 表述方式不同，缺乏映射关系 |
| **测试规则** | 详细测试分类表格（10大类） | 仅提及"测试边界清晰" | 详细程度差异大 |
| **开发流程** | 完整流程图：拆解 → 设计 → 编码 → 测试 → 验证通过 | 无明确流程 | 缺少流程指导 |
| **语言特性** | 针对 React/TypeScript 项目 | 语言无关 | 适用范围不同 |
| **产出要求** | 强制产出文件（tasks.md、architecture.md、test.md） | 无产出要求 | 执行力度不同 |
| **检查清单** | 无 | 有架构健康检查清单 | 互补性强 |

## 二、核心冲突分析

### 2.1 架构原则表述冲突

#### rules 的表述（目标导向）
```
健康的架构设计必须确保以下质量属性：
- 可靠性、可维护性、可扩展性、性能效率
- 可测试性、可观测性、安全性、兼容性、可移植性
```

#### skill 的表述（实现导向）
```
六大设计维度：
1. 设计原则（SOLID、组合优于继承等）
2. 项目结构（按功能分包 vs 按类型分包）
3. 接口与契约（小接口优于大接口）
4. 错误处理（fail-fast vs 容错降级）
5. 数据流与状态管理（单向数据流、不可变数据）
6. 可测试性设计（依赖注入、副作用隔离）
```

#### 冲突点
- **视角不同**：rules 从质量目标出发，skill 从实现手段出发
- **缺乏映射**：没有说明如何通过设计维度实现质量属性
- **容易混淆**：AI 不知道应该遵循哪个标准

### 2.2 测试规则详细程度冲突

#### rules 的测试分类（详细）
```
| 分类 | 测试类型 |
|------|----------|
| 操作间干扰 | 并发冲突、操作中断、状态残留、重复提交 |
| 数据一致性 | 数据同步、数据边界、数据回滚 |
| 边界与极限 | 极限状态、空状态、满载状态 |
| 恢复能力 | 异常恢复、状态持久化、降级处理 |
| 用户预期管理 | 反馈及时性、可撤销性、进度可感知 |
| 渲染与视觉 | 渲染一致性、视觉回归、布局稳定性、响应式适配、暗黑/浅色模式 |
| 交互反馈 | 操作可取消、防误触、反馈时效、键盘可访问性 |
| 性能与流畅度 | 首屏加载、滚动流畅度、输入响应延迟、内存泄漏 |
| 安全与权限 | 越权访问、数据暴露、输入注入、操作权限 |
| 可维护性 | 错误边界、日志可追踪、配置化程度、向后兼容 |
```

#### skill 的测试规则（简略）
```
测试边界清晰：
- 单元测试：单个函数/模块
- 集成测试：模块间交互
- 端到端测试：全系统
```

#### 冲突点
- **详细程度差异巨大**：rules 有10大类详细分类，skill 只有3种测试类型
- **分类维度不同**：rules 按测试场景分类，skill 按测试范围分类
- **容易遗漏**：AI 可能只遵循 skill 的简略规则，忽略 rules 的详细分类

### 2.3 开发流程冲突

#### rules 的完整流程
```
拆解 → 设计 → 编码 → 测试 → 验证通过 → 进入下一步
```

#### skill 的流程
```
无明确流程，只有架构健康检查清单
```

#### 冲突点
- **流程缺失**：skill 缺少明确的开发流程指导
- **执行顺序不明确**：AI 不知道何时使用 skill 的检查清单

### 2.4 语言特性冲突

#### rules 的语言特性
```
针对 React + TypeScript 项目：
- 函数组件 + Hooks
- Zustand 状态管理
- Tailwind CSS
```

#### skill 的语言特性
```
语言无关：
- 适用于任何编程语言（OOP、FP、过程式）
- 需要根据项目语言特性适配
```

#### 冲突点
- **适用范围不同**：rules 针对特定技术栈，skill 通用性强
- **适配成本**：AI 需要额外适配工作

## 三、优化整合方案

### 3.1 整合原则

1. **互补而非替代**：保留两者的优势，形成互补关系
2. **目标与手段结合**：rules 定义目标，skill 提供实现手段
3. **流程与检查结合**：rules 定义流程，skill 提供检查清单
4. **通用与特定结合**：skill 提供通用原则，rules 提供特定约束

### 3.2 优化方案

#### 方案1：分层整合（推荐）

```
.trae/rules/project_rules.md (项目级强制规则)
├── 任务拆解规则
├── 架构设计规则
│   ├── 质量目标（8个质量属性）
│   └── 引用 skill/healthy-architecture（实现手段）
├── 测试规则
│   ├── 测试分类（10大类）
│   └── 引用 skill/test-discipline（测试策略）
└── 开发流程
    ├── 完整流程图
    └── 引用 skill/phased-execution（分阶段执行）

.trae/skill/healthy-architecture/SKILL.md (通用架构规范)
├── 核心原则
├── 六大设计维度（如何实现质量目标）
├── 架构健康检查清单
└── 与 rules 的映射关系
```

#### 方案2：内容合并

将 skill 的内容合并到 rules 中，形成统一规范：
- 优点：避免冲突，易于理解
- 缺点：文件过大，失去通用性

#### 方案3：引用关系

在 rules 中引用 skill，保持独立性：
- 优点：保持通用性，易于维护
- 缺点：需要明确引用关系

### 3.3 具体优化建议

#### 建议1：建立质量属性与设计维度的映射关系

```markdown
## 架构质量属性与设计维度映射

| 质量属性 | 相关设计维度 | 具体实践 |
|----------|-------------|----------|
| 可靠性 | 错误处理、数据流 | fail-fast、容错降级、单向数据流 |
| 可维护性 | 项目结构、接口与契约 | 按功能分包、小接口、向后兼容 |
| 可扩展性 | 设计原则、接口与契约 | 开闭原则、扩展不加改 |
| 性能效率 | 数据流、可测试性 | 不可变数据、副作用隔离 |
| 可测试性 | 可测试性设计 | 依赖注入、副作用隔离 |
| 可观测性 | 错误处理 | 错误传播链、日志可追踪 |
| 安全性 | 接口与契约、错误处理 | 输入校验、业务异常分离 |
| 兼容性 | 接口与契约 | 向后兼容策略、版本化 |
| 可移植性 | 项目结构 | 按功能分包、语言无关原则 |
```

#### 建议2：整合测试规则

```markdown
## 测试规则整合

### 测试分类（来自 rules）
- 操作间干扰、数据一致性、边界与极限、恢复能力
- 用户预期管理、渲染与视觉、交互反馈、性能与流畅度
- 安全与权限、可维护性

### 测试策略（来自 skill）
- 单元测试：覆盖核心逻辑（纯函数、计算、判断）
- 集成测试：覆盖模块间交互（接口契约、数据流）
- 端到端测试：覆盖关键用户路径

### 测试边界（来自 skill）
- 纯逻辑层（无副作用）：可独立单元测试
- IO 层（有副作用）：用集成测试覆盖
```

#### 建议3：明确开发流程与检查清单的关系

```markdown
## 完整开发流程

### 流程步骤（来自 rules）
1. 拆解 → 产出 tasks.md
2. 设计 → 产出 architecture.md
3. 编码 → 实现功能
4. 测试 → 产出 test.md
5. 验证通过 → 进入下一步

### 检查点（来自 skill）
- 每个阶段完成后，对照架构健康检查清单自查
- 编码完成后，对照代码审查清单检查
- 测试完成后，对照测试分类表验证
```

## 四、实施计划

### 4.1 第一阶段：建立映射关系
1. 创建质量属性与设计维度映射表
2. 整合测试规则
3. 明确流程与检查清单的关系

### 4.2 第二阶段：更新文档
1. 更新 `.trae/rules/project_rules.md`
2. 更新 `.trae/skill/healthy-architecture/SKILL.md`
3. 添加交叉引用

### 4.3 第三阶段：验证一致性
1. 检查是否有冲突
2. 确保引用关系正确
3. 验证 AI 理解一致性

## 五、总结

通过分层整合的方式，可以保留两者的优势：
- **rules**：项目级强制规则，明确流程和产出要求
- **skill**：通用架构规范，提供实现手段和检查清单

关键是要建立清晰的映射关系，让 AI 知道：
1. **目标是什么**（rules 的质量属性）
2. **如何实现**（skill 的设计维度）
3. **如何验证**（rules 的测试分类 + skill 的检查清单）
4. **流程是什么**（rules 的开发流程）
