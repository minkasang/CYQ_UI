# 项目术语表

> 本文件记录个人工作台的统一语言。产品语境中的“用户”均指终端使用者，不指开发者。

---

## AI 与智能体

| 术语 | 规范定义 | 备注 |
|------|----------|------|
| AI 模块 | 个人工作台中的通用 AI 聊天模块，主要承载单次对话、普通问答和配置能力。 | 不等同于圆桌工作坊。 |
| Agent / 智能体 | 具有名称、模型、人设、回复参数和可选能力模块的 AI 角色实体。 | 存储在 `agents.json`。 |
| Agent 1v1 对话 | 用户与单个 Agent 的直接对话。 | 根据 ADR 0001，与群聊分离存储。 |
| 智能体群聊 | 多个 Agent 参与同一个聊天空间，按某种 ChatStrategy 回复。 | 是底层互动能力，不等同于产品化工作坊。 |
| ChatStrategy / 聊天策略 | 决定多 Agent 如何发言、并行、串行、冷却、收敛的调度接口。 | 策略应保持小接口、深实现。 |
| AgentModule / 智能体能力模块 | 给 Agent 注入记忆、工具、关系等额外上下文的能力扩展点。 | 目前主要是预留接口。 |

## AI 圆桌工作坊

| 术语 | 规范定义 | 备注 |
|------|----------|------|
| AI 圆桌工作坊 | 面向用户的多 Agent 专家协作流程：用户提出模糊想法，AI 专家组按阶段分析、质疑、收敛并输出可执行方案。 | 产品名，可简称“圆桌工作坊”。 |
| 圆桌会话 | 一次完整的工作坊运行实例，包含输入想法、参与角色、阶段记录、讨论消息和最终产出。 | 可作为群聊的一种模式设计。 |
| 主持人 | 控制流程的 Agent 角色，负责澄清问题、推进阶段、压缩上下文、收敛结论。 | 主持人不等同于普通专家。 |
| 专家角色 | 在圆桌中承担固定视角的 Agent，如产品专家、体验设计师、技术架构师、风险官、执行规划师。 | 角色决定分析责任和 prompt 边界。 |
| 分镜分析 | 每个专家按自己的视角独立观察同一个想法，形成不同“镜头”的分析。 | 对应用户说的“多 AI 镜头互动”。 |
| 反方审查 | 由风险官或反对者角色主动质疑方案，寻找失败点、过度设计、不可落地之处。 | 防止 AI 只会附和。 |
| 结构化产出 | 圆桌结束后生成的可复用结果，包括目标定义、用户路径、MVP 范围、功能清单、实现建议、风险、下一步任务。 | 不是普通聊天摘要。 |
| 独立初稿 | 发散阶段中，各专家在看到其他专家观点之前先独立输出自己的第一版判断。 | 用于降低群体锚定和互相附和。 |
| 发言候选池 | 每个阶段允许发言的专家集合，由阶段规则过滤后再选择实际发言者。 | 比固定轮流更灵活，比自由群聊更可控。 |
| 质量门禁 | 对最终结构化产出按固定标准检查，未达标时触发有限次数重写。 | 不等同于主观评价，要有明确 rubric。 |
| 用户检查点 | 工作坊在关键阶段暂停，让用户确认方向、补充信息或选择偏好。 | 是人机协同，不是简单确认弹窗。 |
| 上下文隔离 | 不同专家只接收当前职责所需的输入，避免无关历史污染判断。 | 独立初稿尤其需要隔离上下文。 |
| 可观测记录 | 记录阶段、发言者、失败、跳过、产物版本、完整性等运行信息。 | 用于调试“为什么这次分析不好”。 |

## 工作坊阶段

| 阶段 | 定义 | 进入条件 | 退出条件 |
|------|------|----------|----------|
| 澄清 | 主持人判断用户想法是否足够明确，并提出必要问题。 | 用户提交想法。 | 信息足够进入分析，或用户选择跳过。 |
| 发散 | 专家组从产品、体验、技术、价值等角度独立分析。 | 澄清完成。 | 每个必要专家至少输出一次有效观点。 |
| 质疑 | 风险官审查方案漏洞，专家可针对质疑补充。 | 发散观点已形成。 | 关键风险被记录或降级处理。 |
| 收敛 | 主持人合并共识，去掉重复和空泛内容。 | 质疑完成。 | 形成明确方案骨架。 |
| 产出 | 系统生成结构化方案，并给出下一步动作。 | 收敛完成。 | 产物可保存、复制或转任务拆解。 |

## 边界约定

| 概念 | 做什么 | 不做什么 |
|------|--------|----------|
| AI 圆桌工作坊 | 把模糊想法转成可执行方案。 | 不替代普通 AI 聊天，不做无限自由闲聊。 |
| 智能体群聊 | 提供多 Agent 消息互动能力。 | 不负责定义产品流程和最终产出格式。 |
| Agent 1v1 对话 | 适合和单个角色深入追问。 | 不承担多角色协作流程。 |
| 每日灵感 | 捕获用户日常灵感；产品 UI 已升级为「人生图谱」，模块 id 仍为 `inspiration`。 | 不直接承担圆桌分析，但可作为圆桌输入来源。 |

## 人生图谱

| 术语 | 规范定义 | 备注 |
|------|----------|------|
| 人生图谱 | “每日灵感”的升级方向，用来长期收集、理解并转化人生中的感悟、价值观、内在模式和行动提醒。 | 产品上层名，默认是私人安静的内在空间，需要时展开成成长工具。 |
| 收集箱 | 人生图谱的低门槛入口，允许用户先保存杂乱的想法、摘录、情绪、原则、提醒或未成型片段。 | 不要求用户一开始分类或结构化。 |
| 洞察卡片 | 从收集箱内容提炼出的更清晰认知，包含为什么重要、关联主题、可能反映的价值观或模式。 | 可以由用户手动填写，后续可接入 AI 辅助提炼。 |
| 内在图谱 | 对洞察卡片中反复出现的主题、价值观、行为模式、情绪线索的聚合视图。 | 用于帮助用户看见长期自我模式。 |
| 原则提醒 | 从重要洞察沉淀出的可反复回看的生活原则或判断准则。 | 不是强制任务，更像人生校准器。 |
| 小行动实验 | 由洞察生成的轻量改变动作，强调下一次遇到类似情境时做一个小变化。 | 不做惩罚式打卡，失败只作为信息。 |
| 漫游模式 | 低压力浏览状态，适合翻看、回忆、感受和连接过去记录。 | 默认体验。 |
| 锚定模式 | 面向改变的结构化状态，围绕价值观、障碍、如果……那么……和小行动实验展开。 | 用户准备改变时再进入。 |
