# 分阶段架构设计与验证

## 核心原则

无论任务是写代码还是写规范方案，必须遵循以下三大原则：

1. **健康的架构**：模块职责清晰、高内聚低耦合、接口定义明确、错误处理完备
2. **分阶段执行**：将一个大型任务拆分为多个独立可验证的阶段，禁止一次性交付全部
3. **每阶段必验证**：每个阶段完成后必须有测试或验证步骤，确认正确后再进入下一阶段

---

## 执行流程

### 第一步：任务分析与阶段拆分

接到任务后，先分析并输出阶段拆分计划，包含：

- **阶段列表**：每个阶段的目标、输入、输出
- **依赖关系**：哪些阶段可以并行，哪些必须串行
- **验证方式**：每个阶段如何验证（单元测试 / 集成测试 / 手动验证 / 编译检查 / lint 检查）
- **架构概览**：整体架构的组件/模块图，明确各模块职责

向用户展示阶段计划，获得确认后再开始执行。

### 第二步：逐阶段执行（每个阶段执行以下循环）

#### 2.1 架构设计

在动手写代码或内容之前，先明确：

- **模块/组件划分**：这个阶段涉及哪些模块？每个模块的职责是什么？
- **接口定义**：模块间的交互接口（函数签名、API、数据格式）是什么？
- **数据流**：数据如何在这些模块间流动？
- **错误处理策略**：异常情况如何处理？
- **依赖关系**：该阶段依赖哪些已有模块？会被哪些后续阶段依赖？

#### 2.2 实现

严格按照架构设计进行实现，遵循以下规则：

- **单一职责**：每个函数/类/模块只做一件事
- **接口先行**：先定义接口，再实现细节
- **最小化变更范围**：只修改与该阶段相关的文件，不做无关重构
- **避免过度工程**：不为未来可能的需求预留扩展点
- **代码可测试性**：确保关键逻辑可被独立测试（依赖注入、纯函数优先）

#### 2.3 测试验证

实现完成后，立即运行验证：

- **编译/语法检查**：确保代码能编译通过，无语法错误
- **Lint 检查**：确保代码风格符合规范
- **单元测试**：为核心逻辑编写或运行测试
- **集成检查**：确保该阶段的变更没有破坏已有功能
- **手动验证**：必要时提供手动验证步骤

**只有验证全部通过后，才能进入下一阶段。**

### 第三步：集成验证

所有阶段完成后，执行端到端集成验证：

- 运行完整测试套件
- 检查各阶段之间的接口衔接
- 确认整体功能符合用户预期

---

## 架构健康检查清单

每个阶段结束时，对照以下清单自查：

| 检查项 | 说明 |
|--------|------|
| 模块职责清晰 | 每个模块的边界明确，没有职责重叠或缺失 |
| 接口稳定 | 对外暴露的接口在后续阶段不需要大改 |
| 依赖方向正确 | 高层模块不依赖低层实现细节 |
| 错误有处理 | 异常路径有明确的处理逻辑，不吞异常 |
| 可测试 | 核心逻辑不依赖全局状态，可单独测试 |
| 无过度设计 | 没有为尚未出现的需求预留复杂性 |

---

## 禁止行为

- ❌ 一次性完成整个大型任务而不分阶段
- ❌ 跳过验证直接进入下一阶段
- ❌ 在一个阶段中修改大量无关文件
- ❌ 没有接口设计就直接写实现
- ❌ 忽略测试失败继续推进
- ❌ 为了"更快完成"而牺牲架构清晰度
- ❌ 验证步骤只说不做

## 阶段拆分参考模板

```
## 阶段计划

### 阶段 1：[名称]
- **目标**：具体要实现什么
- **输入**：依赖的前置条件
- **输出**：交付物（代码文件 / 文档 / 配置）
- **验证**：如何验证该阶段正确
- **预计涉及文件**：xxx

### 阶段 2：[名称]
- **目标**：...
- **输入**：阶段 1 的输出
- **输出**：...
- **验证**：...

### 阶段 N：[名称]
...

### 集成验证
- 运行完整测试套件
- 端到端功能验证
```
