# 测试验证纪律

## 核心原则

1. **每阶段必验证**：代码变更后必须有验证步骤，确认正确后再交付
2. **验证分层**：不同层级用不同类型的测试覆盖
3. **验证不做假**：验证通过意味着真正验证过，不能跳过或降级
4. **自适应技术栈**：命令和工具根据项目实际技术栈选择，不预设特定框架

---

## 验证金字塔

```
          /\
         /端到端\       少量，验证关键用户流程
        /--------\
       /  集成测试 \     中等，验证模块间交互
      /------------\
     /   单元测试    \   大量，验证纯逻辑、边界条件
    /----------------\
```

| 层级 | 占比建议 | 覆盖内容 | 执行频率 |
|------|----------|----------|----------|
| **单元测试** | ~60% | 纯逻辑、工具函数、边界条件、错误路径 | 每次提交 |
| **集成测试** | ~30% | 模块间接口、数据流贯通、外部依赖交互 | 每次 PR |
| **端到端测试** | ~10% | 关键用户操作流程、核心业务闭环 | 每次发布 |

## 验证流程

### 每阶段必须执行的验证步骤（按顺序）

1. **编译 / 语法检查**
   - 代码能编译通过，无语法错误
   - 类型检查通过（如项目有对应工具）
   - **命令检测策略**：先检查项目已有的构建脚本（package.json、Makefile、Cargo.toml、pom.xml、build.gradle 等），运行对应的构建命令；如果没有，使用语言标准命令

2. **Lint 检查**
   - 代码风格符合项目规范
   - 无未使用的变量、导入
   - **命令检测策略**：检查项目已配置的 lint 工具（如 .eslintrc、pyproject.toml、.golangci.yml、checkstyle.xml 等），运行项目自带的 lint 脚本；如果项目未配置 lint，可跳过此步

3. **单元测试**
   - 新增代码的关键路径、边界条件、错误处理必须有测试覆盖
   - 已有测试必须全部通过，不能有退化
   - **命令检测策略**：检查项目测试脚本（如 `npm test`、`cargo test`、`go test ./...`、`pytest`、`mvn test`、`gradle test` 等），运行对应的测试命令

4. **集成检查**
   - 确认该阶段的变更没有破坏已有功能
   - 运行受影响的集成测试

5. **手动验证（必要时）**
   - 无法自动化验证的 UI、交互等
   - 给出明确的验证步骤

**只有以上全部通过后，才能进入下一阶段。**

### 快速命令参考（按技术栈）

| 技术栈 | 构建检查 | Lint 工具 | 测试命令 |
|--------|----------|-----------|----------|
| Node.js | `npm run build` | ESLint / Prettier | `npm test` |
| TypeScript | `tsc --noEmit` | ESLint / Prettier | `npm test` |
| Python | `python -m py_compile` | flake8 / ruff / pylint | `pytest` |
| Go | `go build ./...` | golangci-lint / go vet | `go test ./...` |
| Rust | `cargo build` / `cargo check` | cargo clippy / rustfmt | `cargo test` |
| Java | `mvn compile` / `gradle build` | checkstyle / spotbugs | `mvn test` / `gradle test` |
| C/C++ | `cmake --build` / `make` | cpplint / clang-tidy | `ctest` / `make test` |
| C#/.NET | `dotnet build` | dotnet format / StyleCop | `dotnet test` |
| Ruby | `ruby -c` | rubocop / standard | `rspec` / `rake test` |
| Dart/Flutter | `dart analyze` / `flutter analyze` | dart format | `flutter test` / `dart test` |

## 测试编写规范

### 单元测试要求

- **覆盖率目标**：关键路径尽可能全覆盖，整体覆盖率根据项目阶段设定（早期项目可放宽，生产代码应 >80%）
- **测试独立**：每个测试不依赖其他测试的执行顺序
- **命名清晰**：遵循语言社区的命名惯例（Python: `test_<模块>_<场景>`, Java: `<场景>Test`, JavaScript: `test('<场景>')` 等）
- **AAA 模式**：Arrange（准备）→ Act（执行）→ Assert（断言）

### 必须测试的场景

| 场景 | 说明 |
|------|------|
| **正常路径** | 正常输入，得到预期输出 |
| **边界条件** | 空值、零值、最大值、最小值 |
| **异常路径** | 非法输入、外部依赖失败、超时 |
| **并发安全** | 如果有共享状态，测试并发读写 |

### 集成测试要求

- 验证模块间接口契约是否一致
- 验证数据在各模块间流动的正确性
- 外部依赖（数据库、API）用真实或容器化的环境，非 mock

## 验证门禁

### 新代码的验证门禁

```
  提交代码
     ↓
  编译检查 ──失败──→ 修复后重新提交
     ↓ 通过
  Lint 检查 ──失败──→ 修复后重新提交
     ↓ 通过
  单元测试 ──失败──→ 修复后重新提交
     ↓ 通过
  集成检查 ──失败──→ 修复后重新提交
     ↓ 通过
  ✓ 进入下一阶段
```

### 阶段间验证门禁

```
  阶段 N 完成
     ↓
  阶段 N 的测试全部通过？
     ↓ 是                ↓ 否
  已有测试无退化？        修复 →
     ↓ 是
  ✓ 允许进入阶段 N+1
```

---

## 禁止行为

- ❌ 跳过验证直接进入下一阶段
- ❌ 忽略测试失败继续推进
- ❌ 写了测试但用 skip / 注释掉
- ❌ 仅靠手动测试替代自动化测试
- ❌ 为了覆盖率指标写无意义的测试
- ❌ 集成测试全部用 mock（失去集成测试的意义）
- ❌ 验证步骤只说不做
