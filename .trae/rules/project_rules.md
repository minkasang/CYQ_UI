# AI 执行规则

所有任务的状态和阶段变更实时记录在 `.trae/rules/task_manifest.yaml`，AI 启动时应先读此文件了解全局进度，无需遍历各任务目录。

### 任务清单规则
- 每个任务对应 `共享执行规则文件/{任务名}/`，产出 tasks.md / architecture.md / test.md
- 一次只能 1 个 in_progress
- 中途插新需求时，当前任务变 blocked，新任务为 in_progress，`parent`/`blocked_by` 关联，支持多层嵌套
- 完成后移入 archived 段
- **每次 AI 启动读完 manifest 后，必须校对所有 active 任务状态是否与实际情况一致，不一致则立即更新 manifest**

## 任务拆解规则

注意！大纲类的内容不能依赖记忆，需要产出文件来对照文件大纲来实现，以及实施中有改动时需要及时更新文件，重中之重：绝对不能改蹦原来功能。无论是优化还是重构等，都必须保持系统的功能和性能不下降，不能引入新的问题。

任务拆解不能依赖记忆，需要产出文件实现任务拆解过程，文件放置在共享执行规则文件/任务名称/下的`tasks.md`文件中，文件内容按照以下要求：

多级任务分解：执行任务时必须拆解为多级子任务，拆到每一步可独立完成、可独立验证、可独立回退。拆解层级和步数由任务复杂度动态决定，不固定。每步完成后必须进行测试，通过后进入下一步。

任务拆解需要考虑以下因素：
1. 任务复杂度
2. 任务依赖关系
3. 任务时间限制等等

杜绝以下例子：比如你AI拆了总共20步之后，但还是一口气做完了，其实这个任务分解本来是想要AI知道，你一次对话产出的代码数量是有限制的，如果要实现的功能很大，很难，那么一次对话要跨一个大阶段太难，可能是要分好多次对话来做，比如它分了10大阶段，有的阶段很大，你拆完还是一口气做完了，有限的代码实现十个大阶段，不用想也知道实际上效果不怎么滴，所以用户目的很明确，不要求你很快，必须要保证质量，保证架构代码的质量以及完整性，健康性，可以多次对话来做。

## 架构设计规则

架构设计不能依赖记忆，需要产出文件实现架构设计过程，文件放置在共享执行规则文件/任务名称/下的`architecture.md`文件中。

记住架构设计必须要健康，遵守以下架构设计原则：

### 质量目标（必须确保）

健康的架构设计必须确保以下质量属性：

| 质量属性 | 说明 | 相关设计维度 |
|----------|------|-------------|
| **可靠性** | 系统稳定运行，错误可恢复 | 错误处理、数据流 |
| **可维护性** | 代码易于理解、修改、扩展 | 项目结构、接口与契约 |
| **可扩展性** | 新增功能时修改最小化 | 设计原则、接口与契约 |
| **性能效率** | 响应快速，资源占用合理 | 数据流、可测试性 |
| **可测试性** | 核心逻辑可独立测试 | 可测试性设计 |
| **可观测性** | 系统状态可监控、可追踪 | 错误处理、日志 |
| **安全性** | 数据安全，权限可控 | 接口与契约、错误处理 |
| **兼容性** | 向后兼容，版本平滑升级 | 接口与契约 |
| **可移植性** | 跨平台、跨环境部署 | 项目结构 |

### 实现手段（引用 skill）

为实现上述质量目标，必须遵循 `.trae/skill/healthy-architecture/SKILL.md` 中的六大设计维度：

1. **设计原则**：SOLID 原则、组合优于继承、最少知识原则
2. **项目结构**：按功能分包、三次法则、目录命名规范
3. **接口与契约**：小接口优于大接口、参数明确、向后兼容策略
4. **错误处理**：fail-fast vs 容错降级、错误传播链、业务异常与技术异常分离
5. **数据流与状态管理**：单向数据流、不可变数据优先、状态归属原则
6. **可测试性设计**：依赖注入、副作用隔离、测试边界清晰

**重要**：在架构设计时，必须明确说明如何通过设计维度实现质量目标。

### 通用架构原则

- **模块化与分层**：按功能模块组织代码，每层职责明确（表现层、逻辑层、数据层、服务层）
- **明确边界与依赖方向**：外层可依赖内层，内层绝不依赖外层，依赖方向向内收敛
- **状态分类管理**：状态分为本地UI状态、共享客户端状态、服务端状态三类，分别用不同工具管理
- **稳定的公共API**：每个模块只暴露必要接口，内部实现细节对外不可见
- **测试策略分层**：单元测试覆盖核心逻辑，集成测试覆盖模块间交互，E2E测试覆盖关键用户路径

## 测试规则

写完代码后，必须要进行测试，确保系统的稳定性和可靠性，测试内容不能依赖记忆，需要产出文件实现测试过程，文件放置在共享执行规则文件/任务名称/下的`test.md`文件中。

### 测试分类（按场景）

| 分类 | 测试类型 | 说明 |
|------|----------|------|
| **操作间干扰** | 并发冲突、操作中断、状态残留、重复提交 | 验证多操作并发时的数据一致性 |
| **数据一致性** | 数据同步、数据边界、数据回滚 | 验证数据的完整性和一致性 |
| **边界与极限** | 极限状态、空状态、满载状态 | 验证极端情况下的系统行为 |
| **恢复能力** | 异常恢复、状态持久化、降级处理 | 验证系统的容错和恢复能力 |
| **用户预期管理** | 反馈及时性、可撤销性、进度可感知 | 验证用户体验的流畅性 |
| **渲染与视觉** | 渲染一致性、视觉回归、布局稳定性、响应式适配、暗黑/浅色模式 | 验证UI的正确性和一致性 |
| **交互反馈** | 操作可取消、防误触、反馈时效、键盘可访问性 | 验证交互的友好性 |
| **性能与流畅度** | 首屏加载、滚动流畅度、输入响应延迟、内存泄漏 | 验证系统的性能表现 |
| **安全与权限** | 越权访问、数据暴露、输入注入、操作权限 | 验证系统的安全性 |
| **可维护性** | 错误边界、日志可追踪、配置化程度、向后兼容 | 验证系统的可维护性 |
| **实际效果** | 实际的功能是否生效 | 验证系统的功能是否符合预期 |

### 测试策略（引用 skill）

根据 `.trae/skill/healthy-architecture/SKILL.md` 的测试边界划分：

| 测试类型 | 范围 | 测试内容 | 覆盖目标 |
|----------|------|----------|----------|
| **单元测试** | 单个函数/模块 | 纯逻辑：计算、判断、转换 | 核心逻辑正确性 |
| **集成测试** | 模块间交互 | 接口契约、数据流贯通 | 模块协作正确性 |
| **端到端测试** | 全系统 | 用户操作流程 | 关键路径可用性 |

### 测试执行原则

1. **副作用隔离**：纯逻辑（无副作用）用单元测试，IO 操作（有副作用）用集成测试
2. **测试优先级**：核心业务逻辑优先，边缘场景次之
3. **测试覆盖**：每个测试分类至少覆盖一个测试类型
4. **测试报告**：必须在 test.md 中记录测试结果和问题

## 代码质量标准

代码必须符合以下标准：
- 代码必须符合项目的技术规范和编码标准
- 代码必须符合项目的功能需求和性能指标
- 代码必须符合项目的安全要求和隐私保护
- 代码必须符合项目的可维护性和可扩展性
- 代码必须符合项目的可部署性和可监控性

在保证代码质量的前提下，务必保持代码简洁、清晰、易读，不要出现重复代码、冗余逻辑、低效算法等性能问题。

## 完整开发流程

```
拆解 → 设计 → 编码 → 测试 → 验证通过 → 进入下一步
```

这样才是一个完整的开发流程，确保了代码的质量和系统的稳定性。最终输出测试通过报告文件。

测试报告直接在共享执行规则文件/任务名称/下的`test.md`文件中。

一个任务的`tasks.md`、`architecture.md`、`test.md`等的文件的【任务名称】必须是唯一且一致的，否则会搞混导致找不到正确的文件。

### 流程详解

#### 1. 拆解阶段
- **产出文件**：`tasks.md`
- **要求**：多级任务分解，每步可独立完成、可独立验证、可独立回退
- **检查点**：任务拆解是否合理，是否考虑了依赖关系

#### 2. 设计阶段
- **产出文件**：`architecture.md`
- **要求**：明确架构设计，说明如何通过设计维度实现质量目标
- **检查点**：对照 `.trae/skill/healthy-architecture/SKILL.md` 的架构健康检查清单自查

#### 3. 编码阶段
- **产出文件**：代码实现
- **要求**：遵循编码规范，保持代码简洁清晰
- **检查点**：对照 `.trae/skill/healthy-architecture/SKILL.md` 的禁止行为清单检查

#### 4. 测试阶段
- **产出文件**：`test.md`
- **要求**：按照测试分类表格进行全面测试
- **检查点**：确保每个测试分类至少覆盖一个测试类型

#### 5. 验证通过阶段
- **产出文件**：测试通过报告
- **要求**：所有测试通过，无遗留问题
- **检查点**：对照测试执行原则验证

### 检查清单引用

在每个阶段完成后，必须对照以下检查清单：

1. **架构健康检查清单**（来自 `.trae/skill/healthy-architecture/SKILL.md`）
   - 模块职责清晰
   - 接口稳定
   - 依赖方向正确
   - 错误有处理
   - 可测试
   - 无过度设计
   - 副作用隔离

2. **禁止行为清单**（来自 `.trae/skill/healthy-architecture/SKILL.md`）
   - ❌ 没有接口设计就直接写实现
   - ❌ 为了"更快完成"而牺牲架构清晰度
   - ❌ 深层继承链（>3 层）或用继承复用行为
   - ❌ 链式调用穿过 3 个以上对象
   - ❌ 第一次出现重复就急着重构抽象
   - ❌ 吞异常不传播上下文
   - ❌ 混放业务异常和技术异常
   - ❌ 状态放在共享层但只被一个模块使用
