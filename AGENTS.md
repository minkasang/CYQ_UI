## USER_PROFILE.md（用户画像）

项目根目录的 `USER_PROFILE.md`（如果存在）记录了 AI 在与用户长期协作中总结的用户偏好。AI 在以下场景**必须**先读取它：

- 每次对话开始时
- 做设计决策（风格/配色/交互）时
- 选择方案 A/B 需要倾向性判断时

**内容示例**：喜欢的 UI 风格、讨厌的交互模式、常用术语习惯、代码组织偏好等。

AI 在每次协作中发现新的用户偏好时，应主动在对话中总结并更新 `USER_PROFILE.md`。如果文件不存在，在第一次对话结束后创建它。维护规则：只记录反复出现的模式，不记录偶然的一时偏好。

## 项目上下文

### CONTEXT.md（项目术语表）

项目根目录的 `CONTEXT.md`（如果存在）是项目的**统一语言（Ubiquitous Language）**——定义了项目中所有特定领域术语的唯一规范表达。AI 在以下场景**必须**先读取它：

- 理解用户需求时
- 命名变量/函数/文件时
- 写测试描述时
- 探索代码库时
- 任何涉及项目特定概念的沟通时

如果 `CONTEXT.md` 不存在，在每个任务开始前询问用户是否需要建立。维护规则参考 `.reasonix/skills/domain-modeling/SKILL.md`。

### ADR（架构决策记录）

`docs/adr/` 目录存放不可轻易逆转的架构决策。AI 在以下场景**必须**先检查相关 ADR：

- 提出架构变更时
- 质疑已有设计时
- 架构体检（improve-codebase-architecture）时

不要重新争论已有 ADR 记录的决策——除非当前摩擦足够真实到值得重新打开它。

## 启动前对齐协议

**在进入任务拆解之前**，AI 必须与用户完成对齐。原则来自《程序员修炼之道》："没有人确切知道自己想要什么。"

### 轻量对齐（小任务，≤2 文件变更）

AI 用 1-2 个关键问题确认范围和预期，然后直接进入拆解。

### 完整盘问（中大型任务，>2 文件或新模块）

使用 `.reasonix/skills/grilling/SKILL.md` 中的盘问流程：

1. **理解意图**：AI 概括理解的范围，让用户确认
2. **遍历决策树**：逐个问题（一次只问一个），走完设计空间的每个分支
3. **共识总结**：输出要做什么/不做什么/关键决策/验收标准

盘问过程中，当术语确定时即时更新 `CONTEXT.md`，不可逆转的决策提议创建 ADR。

**只有盘问完成、用户确认共识总结后，才能进入任务拆解。**

所有任务的状态和阶段变更实时记录在 `.reasonix/task_manifest.yaml`，AI 启动时应先读此文件了解全局进度，无需遍历各任务目录。

### 任务清单规则
- 每个任务对应 `共享执行规则文件/{任务名}/`，产出 tasks.md / architecture.md / test.md
- **任务目录命名**：`YYYY-MM-DD-HHMM_{任务名}`（如 `2026-06-20-1202_设置页设计改造`）。时间为创建任务目录的时刻，精确到分钟，不写秒。同一分钟内创建多个文件不会产生时间不一致
- 一次只能 1 个 in_progress
- 中途插新需求时，当前任务变 blocked，新任务为 in_progress，`parent`/`blocked_by` 关联，支持多层嵌套
- 完成后移入 archived 段
- **每次 AI 启动读完 manifest 后，必须校对所有 active 任务状态是否与实际情况一致，不一致则立即更新 manifest**

## 任务拆解规则

注意！大纲类的内容不能依赖记忆，需要产出文件来对照文件大纲来实现，以及实施中有改动时需要及时更新文件，重中之重：绝对不能改蹦原来功能。无论是优化还是重构等，都必须保持系统的功能和性能不下降，不能引入新的问题。

任务拆解不能依赖记忆，需要产出文件实现任务拆解过程，文件放置在共享执行规则文件/任务名称/下的`tasks.md`文件中，文件内容按照以下要求：

多级任务分解：执行任务时必须拆解为多级子任务，拆到每一步可独立完成、可独立验证、可独立回退。**优先使用垂直切片（每片贯穿所有技术层，schema→逻辑→UI→测试），避免按技术层横切（如先写所有 Model 再写所有 API）——垂直切片天然满足独立验证要求，技术层横切则无法在单步内验证。** 拆解层级和步数由任务复杂度动态决定，不固定。每步完成后必须进行测试，通过后进入下一步。

任务拆解需要考虑以下因素：
1. 任务复杂度
2. 任务依赖关系
3. 任务时间限制等等

杜绝以下例子：比如你AI拆了总共20步之后，但还是一口气做完了，其实这个任务分解本来是想要AI知道，你一次对话产出的代码数量是有限制的，如果要实现的功能很大，很难，那么一次对话要跨一个大阶段太难，可能是要分好多次对话来做，比如它分了10大阶段，有的阶段很大，你拆完还是一口气做完了，有限的代码实现十个大阶段，不用想也知道实际上效果不怎么滴，所以用户目的很明确，不要求你很快，必须要保证质量，保证架构代码的质量以及完整性，健康性，可以多次对话来做。

## 架构设计规则

架构设计不能依赖记忆，需要产出文件实现架构设计过程，文件放置在共享执行规则文件/任务名称/下的`architecture.md`文件中。

记住架构设计必须要健康，遵守以下架构设计原则：

### 质量目标（必须确保）

健康的架构设计必须确保以下质量属性：

| 质量属性 | 说明 | 相关设计维度 |
|----------|------|-------------|
| **可靠性** | 系统稳定运行，错误可恢复 | 错误处理、数据流 |
| **可维护性** | 代码易于理解、修改、扩展 | 项目结构、接口与契约 |
| **可扩展性** | 新增功能时修改最小化 | 设计原则、接口与契约 |
| **性能效率** | 响应快速，资源占用合理 | 数据流、可测试性 |
| **可测试性** | 核心逻辑可独立测试 | 可测试性设计 |
| **可观测性** | 系统状态可监控、可追踪 | 错误处理、日志 |
| **安全性** | 数据安全，权限可控 | 接口与契约、错误处理 |
| **兼容性** | 向后兼容，版本平滑升级 | 接口与契约 |
| **可移植性** | 跨平台、跨环境部署 | 项目结构 |

### 实现手段（引用 skill）

为实现上述质量目标，必须遵循 `.reasonix/skills/healthy-architecture/SKILL.md` 中的设计维度，尤其关注**深度模块设计**（1.4 节）：用小接口隐藏大实现，接口即测试面，删除测试验证模块价值：

1. **设计原则**：SOLID 原则、组合优于继承、最少知识原则、深度模块设计
2. **项目结构**：按功能分包、三次法则、目录命名规范
3. **接口与契约**：小接口优于大接口、参数明确、向后兼容策略
4. **错误处理**：fail-fast vs 容错降级、错误传播链、业务异常与技术异常分离
5. **数据流与状态管理**：单向数据流、不可变数据优先、状态归属原则
6. **可测试性设计**：依赖注入、副作用隔离、测试边界清晰

**重要**：在架构设计时，必须明确说明如何通过设计维度实现质量目标。

### 通用架构原则

- **模块化与分层**：按功能模块组织代码，每层职责明确（表现层、逻辑层、数据层、服务层）
- **明确边界与依赖方向**：外层可依赖内层，内层绝不依赖外层，依赖方向向内收敛
- **状态分类管理**：状态分为本地UI状态、共享客户端状态、服务端状态三类，分别用不同工具管理
- **稳定的公共API**：每个模块只暴露必要接口，内部实现细节对外不可见
- **测试策略分层**：单元测试覆盖核心逻辑，集成测试覆盖模块间交互，E2E测试覆盖关键用户路径
- **架构决策记录**：不可轻松逆转的架构决策必须写入 `docs/adr/`，格式参考 `.reasonix/skills/domain-modeling/SKILL.md`
- **定期架构体检**：建议每隔几天运行 `improve-codebase-architecture`，发现浅度模块并深化

## 测试规则

写完代码后，必须要进行测试，确保系统的稳定性和可靠性，测试内容不能依赖记忆，需要产出文件实现测试过程，文件放置在共享执行规则文件/任务名称/下的`test.md`文件中。

### 测试分类（按场景）

| 分类 | 测试类型 | 说明 |
|------|----------|------|
| **操作间干扰** | 并发冲突、操作中断、状态残留、重复提交 | 验证多操作并发时的数据一致性 |
| **数据一致性** | 数据同步、数据边界、数据回滚 | 验证数据的完整性和一致性 |
| **边界与极限** | 极限状态、空状态、满载状态 | 验证极端情况下的系统行为 |
| **恢复能力** | 异常恢复、状态持久化、降级处理 | 验证系统的容错和恢复能力 |
| **用户预期管理** | 反馈及时性、可撤销性、进度可感知 | 验证用户体验的流畅性 |
| **渲染与视觉** | 渲染一致性、视觉回归、布局稳定性、响应式适配、暗黑/浅色模式 | 验证UI的正确性和一致性 |
| **交互反馈** | 操作可取消、防误触、反馈时效、键盘可访问性 | 验证交互的友好性 |
| **性能与流畅度** | 首屏加载、滚动流畅度、输入响应延迟、内存泄漏 | 验证系统的性能表现 |
| **安全与权限** | 越权访问、数据暴露、输入注入、操作权限 | 验证系统的安全性 |
| **可维护性** | 错误边界、日志可追踪、配置化程度、向后兼容 | 验证系统的可维护性 |
| **实际效果** | 实际的功能是否生效 | 验证系统的功能是否符合预期 |

### 测试策略（引用 skill）

根据 `.reasonix/skills/healthy-architecture/SKILL.md` 的测试边界划分：

| 测试类型 | 范围 | 测试内容 | 覆盖目标 |
|----------|------|----------|----------|
| **单元测试** | 单个函数/模块 | 纯逻辑：计算、判断、转换 | 核心逻辑正确性 |
| **集成测试** | 模块间交互 | 接口契约、数据流贯通 | 模块协作正确性 |
| **端到端测试** | 全系统 | 用户操作流程 | 关键路径可用性 |

### 测试执行原则

1. **副作用隔离**：纯逻辑（无副作用）用单元测试，IO 操作（有副作用）用集成测试
2. **测试优先级**：核心业务逻辑优先，边缘场景次之
3. **测试覆盖**：每个测试分类至少覆盖一个测试类型
4. **测试报告**：必须在 test.md 中记录测试结果和问题

## 调试规则

遇到 bug 时，不得随意猜测原因。必须遵循 `.reasonix/skills/diagnosing-bugs/SKILL.md` 的 6 阶段科学调试法：

1. **建立反馈环**：构建一个能捕获该 bug 的紧致红/绿信号（测试、curl、脚本等）——这是整个方法的核心
2. **复现并最小化**：缩小到仍能变红的最小场景
3. **提出假设**：生成 3-5 个可证伪的假设，排序后展示给用户
4. **定位**：一次只改一个变量，用定向日志（带唯一前缀标签）或调试器
5. **修复 + 回归测试**：先写回归测试（需有正确的缝），再修复
6. **清理 + 复盘**：移除调试代码，问"什么能防止这个 bug"→改进架构

**核心铁律**：没有红/绿反馈环，绝不进入假设阶段。盯着代码看再多也没用。

## 架构体检

建议每隔几天运行 `.reasonix/skills/improve-codebase-architecture/SKILL.md`：

1. 有机探索代码库，感受摩擦点
2. 生成可视化 HTML 报告（Tailwind + Mermaid）
3. 对用户选中的候选项进行盘问式深化

这能系统化地发现浅度模块、紧耦合、测试盲区——防止代码库在 AI 加速开发中腐化。

## 代码质量标准

代码必须符合以下标准：
- 代码必须符合项目的技术规范和编码标准
- 代码必须符合项目的功能需求和性能指标
- 代码必须符合项目的安全要求和隐私保护
- 代码必须符合项目的可维护性和可扩展性
- 代码必须符合项目的可部署性和可监控性

在保证代码质量的前提下，务必保持代码简洁、清晰、易读，不要出现重复代码、冗余逻辑、低效算法等性能问题。

## 完整开发流程

```
对齐 → 规范 → 拆解 → 设计 → 编码 → 测试 → 验证通过 → 进入下一步
```

这样才是一个完整的开发流程，确保了代码的质量和系统的稳定性。最终输出测试通过报告文件。

测试报告直接在共享执行规则文件/任务名称/下的`test.md`文件中。

一个任务的`tasks.md`、`architecture.md`、`test.md`等的文件的【任务名称】必须是唯一且一致的，否则会搞混导致找不到正确的文件。

### 功能设计规范编写规则

每开发一个新模块或新功能，**必须**在 `个人工作台设计规范/{模块名}/` 下编写 `功能设计规范.md`。该文件与 `architecture.md` 不冲突：

| 文件 | 回答什么问题 | 谁用 |
|------|-------------|------|
| `功能设计规范.md` | 做什么功能、用户怎么操作、长什么样、各种状态怎么处理 | 回顾功能、写测试用例、新 AI 了解模块 |
| `architecture.md` | 代码怎么组织、接口怎么定义、数据怎么流 | 编码实现、重构 |

**功能设计规范.md 必须包含**：
- **功能清单**（能做什么、不能做什么、明确的边界）
- 用户路径（从头到尾的操作流程）
- 布局草图（ASCII art 或文字描述每个区域放什么）
- 状态覆盖（空状态/正常态/错误态/极端态各长什么样）
- 视觉参考（参考了哪个 Skill 或 DESIGN.md 的什么风格）

**位置**：`个人工作台设计规范/{模块名}/功能设计规范.md`

**更新规则**：功能变更时同步更新，不得只改代码不改规范。

### 流程详解

#### 0. 对齐阶段
- **产出文件**：共识总结（在对话中输出，不单独建文件） + 更新 `CONTEXT.md`（如有新术语）
- **要求**：使用 `grilling` skill 进行盘问，遍历决策树每个分支
- **检查点**：用户确认共识总结，所有模糊点已解决
- **门禁**：没有共识总结，不允许进入拆解

#### 1. 拆解阶段
- **产出文件**：`tasks.md`
- **要求**：多级任务分解，每步可独立完成、可独立验证、可独立回退
- **检查点**：任务拆解是否合理，是否考虑了依赖关系

#### 2. 设计阶段
- **产出文件**：`architecture.md`
- **要求**：明确架构设计，说明如何通过设计维度实现质量目标
- **检查点**：对照 `.reasonix/skills/healthy-architecture/SKILL.md` 的架构健康检查清单自查

#### 3. 编码阶段
- **产出文件**：代码实现
- **要求**：遵循编码规范，保持代码简洁清晰
- **检查点**：对照 `.reasonix/skills/healthy-architecture/SKILL.md` 的禁止行为清单检查

#### 4. 测试阶段
- **产出文件**：`test.md`
- **要求**：按照测试分类表格进行全面测试
- **检查点**：确保每个测试分类至少覆盖一个测试类型

#### 5. 验证通过阶段
- **产出文件**：测试通过报告
- **要求**：所有测试通过，无遗留问题
- **检查点**：对照测试执行原则验证

### 检查清单引用

在每个阶段完成后，必须对照以下检查清单：

0. **对齐检查清单**（来自 `.reasonix/skills/grilling/SKILL.md`）
   - 共识总结已输出且用户确认
   - 所有模糊点已解决
   - 术语已更新到 CONTEXT.md（如有新术语）
   - 不可逆决策已提议 ADR

1. **架构健康检查清单**（来自 `.reasonix/skills/healthy-architecture/SKILL.md`）
   - 模块职责清晰
   - 接口稳定
   - 接口深度（小接口深实现，不是大接口薄实现）
   - 依赖方向正确
   - 错误有处理
   - 可测试（接口即测试面）
   - 无过度设计（无单一适配器的假想缝）
   - 副作用隔离

2. **禁止行为清单**（来自 `.reasonix/skills/healthy-architecture/SKILL.md`）
   - ❌ 没有接口设计就直接写实现
   - ❌ 为了"更快完成"而牺牲架构清晰度
   - ❌ 深层继承链（>3 层）或用继承复用行为
   - ❌ 链式调用穿过 3 个以上对象
   - ❌ 第一次出现重复就急着重构抽象
   - ❌ 吞异常不传播上下文
   - ❌ 混放业务异常和技术异常
   - ❌ 状态放在共享层但只被一个模块使用

3. **调试检查清单**（来自 `.reasonix/skills/diagnosing-bugs/SKILL.md`）
   - 反馈环已构建且能变红
   - 已生成 3-5 个可证伪假设
   - 回归测试在正确的缝上
   - 所有调试代码已清理
