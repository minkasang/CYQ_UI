# 什么是 API？—— 给非程序员的解释

## 一句话

**API 就是"叫别人帮你干活"的窗口。**

---

## 餐厅比喻

你去餐厅吃饭：

```
你（浏览器看板）        服务员（API）          厨房（Python后端）
     │                      │                      │
     │ "来一份蛋炒饭"        │                      │
     │─────────────────────>│                      │
     │                      │  "蛋炒饭一份！"       │
     │                      │─────────────────────>│
     │                      │                      │ 炒菜中...
     │                      │     🍳                │
     │                      │<─────────────────────│
     │     🍚                │                      │
     │<─────────────────────│                      │
     │                      │                      │
```

- **你** = 浏览器上的看板页面
- **服务员** = API（帮你传话、端菜）
- **厨房** = Python 后端（真正干活的人）

你不直接进厨房炒菜（浏览器不能直接读写硬盘），而是告诉服务员你要什么，服务员去厨房帮你拿。

---

## 真实例子：看板读文件

以前（无 API）：
```
浏览器 → 读取技术规范/感知系统.md → ❌ 浏览器说：我不能读你硬盘
```

以后（有 API）：
```
浏览器 → fetch('/api/read?file=感知系统') → Python → 读文件 → 返回内容 → 浏览器显示 ✅
```

**API 就是中间那个传话的人。**

---

## 三个核心概念

| 概念 | 餐厅比喻 | 看板例子 |
|------|---------|---------|
| **请求 (Request)** | "来一份蛋炒饭" | 浏览器说 "把感知系统规范的内容给我" |
| **处理 (Process)** | 厨房炒菜 | Python 打开文件、读取内容 |
| **响应 (Response)** | 端上来的蛋炒饭 | 返回 md 文件的文字内容 |

---

## URL 就是菜单

你看到的 API 地址，本质就是菜单上的菜名：

```
/api/read?file=感知系统规范     → "读取感知系统规范"
/api/write?file=任务清单          → "写入任务清单"
/api/list?dir=技术规范            → "列出技术规范目录下所有文件"
```

问号 `?` 后面的是你的具体要求（要哪个文件？要什么目录？）。

---

## 为什么需要它？

| 没有 API | 有 API |
|---------|--------|
| 数据写死在 JS 文件里 | 数据从硬盘实时读取 |
| 改一个任务要改 3 个文件 | 改一次 JSON 就完事 |
| 浏览器不能读本地文件 | 浏览器通过 API 间接读写 |
| 数据不会自动更新 | 刷新页面就看到最新数据 |
